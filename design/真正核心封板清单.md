# 真正核心封板清单

## 目标

本清单是具体游戏业务和命令组件开始建设前的唯一验收入口。

真正核心必须完整提供跨玩法共同依赖的机制、持久化和结构化事实。它不能出现宗门、仙城、纳戒、
探险、首领等具体产品名称，也不能等待某个组件开工后再临时补公共能力。

## 固定分层

```text
launch / message
        ^
game/cmd                  命令、参数、权限、展示
        ^
游戏次核心                具体世界规则和玩法编排
        ^
game/core                 真正核心
        ^
SQLite 与外部基础设施
```

依赖只能自上而下。真正核心不能导入游戏次核心、命令、消息协议或通信驱动器。命令组件不能直接
改数据库、余额、库存、行动、活动或社会关系。

任何旧业务如果无法映射到表中已有真正核心能力，必须先修订本清单，不能把公共机制塞入组件。

## 第一阶段：持久化闭环

- [x] 账号、外部身份、身份凭据和冲突记录拥有关系表、唯一约束与原子服务。
- [x] 外部身份数据库键使用服务端 HMAC，日志和普通查询不保存明文 OpenID。
- [x] 行动开始、完成、取消、中断和领取可以并发防重并在重启后恢复。
- [x] 装配状态与库存位置在同一个事务中提交。
- [x] 库存、账本及业务契约可以在同一个外层工作单元中原子提交。
- [x] 所有持久聚合都经过编解码、CAS、损坏检测和重启恢复测试。

## 第二阶段：跨玩法领域

- [x] 掉落与保底：版本化抽取、可落空、权重、独立判定、必得、数量、排除和保底。
- [x] 世界空间：任意世界空间、地点、连接或坐标、存在体、移动、占用和范围查询。
- [x] 区域状态：类型化共享状态、容量、贡献、衰减和阈值，不使用万能键值袋。
- [x] 交换契约：创建、冻结、接受、成交、取消、过期及库存账本原子结算。
- [x] 活动实例：开放、参与、贡献、关闭、排名冻结、资格生成和结算防重。
- [x] 组织与关系：组织、成员、角色、权限、请求、双向或有向关系及并发版本。
- [x] 长期事实：追加式事实记录、投影检查点、幂等重建和保留策略。
- [x] 排名与通知：版本化排名快照、稳定并列规则、通知去重、已读状态和动作语义。

## 封板硬门槛

1. `game/core` 的 AST 依赖巡检通过，不含具体产品模块和命令依赖。
2. 每种持久状态都有空库初始化、写入、并发冲突、幂等重放、重启恢复和损坏拒绝测试。
3. 探险、交易、活动、组织和异步通知各有一个不依赖命令组件的跨系统场景测试。
4. 随机结果必须由 `RuleContext` 驱动并保存版本与审计凭据，禁止读取模块级随机源。
5. 读取状态、排行榜和通知不能顺手推进或修改业务状态。
6. 所有资产变化只能通过库存、账本、角色、武器或奖励事务完成。
7. Outbox 只负责可靠投递；长期事实、投影和通知各有明确的数据所有者。
8. 全量测试、编译、数据库结构校验和真实应用生命周期测试全部通过。
9. 本清单全部完成前，具体游戏次核心和正式命令组件保持冻结。
10. 随机物品保存内容指纹、生成器版本、全部判定和滚值，恢复时必须能重新估值并拒绝漂移。
11. 单件品质只看内在价值；套装和配装协同只进入整套价值与边际价值，不能反向抬高单件品质。
12. 生成武器必须与库存实例在同一奖励事务落库，多套配装必须与库存位置在同一事务切换。

## 不进入真正核心

- 具体物品、武器、敌人、地点、宗门和活动内容。
- 修仙等级、税率、价格、掉落概率、首领难度和奖励数量。
- 玩家可见名称、Markdown、按钮、命令和网页。
- 运营活动配置、人工审批和管理后台。
- `xiuxian3` 表结构兼容和历史数据迁移脚本。

这些内容只能在真正核心封板后，由游戏次核心、内容包、命令组件或独立迁移工具承接。
