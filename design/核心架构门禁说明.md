# 核心架构门禁

`test/core_governance_test.py` 把核心演进边界变成每次都能重复执行的自动检查。规则不依赖开发者记忆，
也不依赖某个具体玩法。

## 数据写入边界

- `game/core/persistence/` 是 SQLite、快照仓库和工作单元的唯一实现层。
- `auto/` 可以创建数据库并完成应用装配。
- 测试可以直接建立临时数据库。
- `game/cmd/`、玩法组件、`game/core/gameplay/` 和 `game/core/account/` 不得导入 SQLite 或
  `game.core.persistence`。正式组件只能调用已经组装好的应用服务，不能读取或修改数据库快照。

这条规则让命令展示、领域规则和数据提交保持三个边界。涉及两个以上领域的原子写入，应新增
`game/core/persistence/` 协调服务，而不是让一个领域拥有另一个领域的仓库。

## 状态所有权

领域聚合的 `*State` 只拥有本领域状态。武器、装备持有类型化的随机物品实例状态是当前明确批准的
组合；其他跨领域状态只能出现在经过审计的执行结果、结算快照或计划对象里。

门禁使用精确到“文件、类、字段、状态类型”的清单。新增组合不能通过放行整个目录解决；需要先确认：

1. 该对象只是一次调用的不可变结果或候选快照，不是新的数据所有者。
2. 原领域仍然独立校验并修改自己的状态。
3. 多领域提交由持久化协调服务原子完成。

队伍是标准示例：队伍只管理成员；社会领域管理邀请；战斗管理阵营；持久化服务协调邀请接受与入队。

## 动态字段

`Any`、`object`、`Mapping[..., object]` 等万能载荷只允许出现在已审计的扩展边界，例如领域事实、
内容定义、通知载荷和奖励明细。门禁冻结当前字段级清单，新增位置会立即失败。

新功能优先新增具名值对象、枚举、联合类型或独立组件定义。只有载荷确实跨内容包扩展，并且消费方具备
明确校验器时，才登记新的动态字段。不得用 `metadata` 或可空字段临时粘合两个领域。

## 底座证据

每个 `*_FOUNDATION_VERSION` 必须同时具备：

- 合法的 `name.foundation.vN` 版本号。
- 同目录中文说明文档。
- 一个直接可运行并断言该版本的 `test/*_test.py`。

新增底座时需要同时登记测试文件。删除字段或组合点时，失效的白名单也会导致测试失败，避免僵尸例外
长期留在架构中。

## 本地与 CI 使用

本地执行：

```powershell
.venv\Scripts\python.exe -X utf8 -B test\core_governance_test.py
```

CI 可以设置比较基线，额外检查核心改动是否同时包含测试、中文文档和公共版本升级：

```powershell
$env:CORE_GUARD_BASE_REF = "origin/main"
.venv\Scripts\python.exe -X utf8 -B test\core_governance_test.py
```

未设置 `CORE_GUARD_BASE_REF` 时不执行 Git 差异检查，因此本地全量测试不依赖远端分支。

## 核心扩展流程

1. 先判断能力属于现有领域、跨领域协调，还是新的独立领域。
2. 为领域契约和失败语义写测试，再实现规则。
3. 跨领域只传稳定 ID、命令和类型化结果，不转交状态所有权。
4. 同步中文说明；公开入口变化时升级公共核心版本。
5. 运行治理门禁、架构门禁和全部底座测试。

因此核心以后仍可增长，但增长形式是新增标准接口和组合点，而不是让具体组件反向塑造核心内部状态。
