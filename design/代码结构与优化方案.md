# 万象行纪代码结构与优化方案

这份文档是现有功能的整理规则，不是新玩法清单。优化以行为不变、边界更清楚、扩展更容易为先。

## 稳定分层

```text
message/                 协议无关的消息模型与构造器
launch/                  驱动器、生命周期、路由与运行时
game/app.py              唯一装配根和对外服务门面
game/core/               通用底座：账户、规则执行、持久化
game/content/            正式内容与世界投影
game/rules/              纯规则与战斗模拟
game/features/           业务事务、存储边界、奖励结算
game/cmd/<中文组件>/      用户命令注册与展示编排
```

依赖方向只能向下：命令调用业务，业务调用规则和底座；规则不读取数据库，内容不反向调用业务。世界皮肤只负责展示投影，真实世界和坐标绑定由世界内容层提供。

## 文件规划

- `game/app.py` 是唯一组合根。它可以很长，但只允许出现依赖组装、服务门面和生命周期状态，不承载业务规则。
- 组件目录保留中文，组件内部模块使用 ASCII，并按用户流程或明确职责拆分。例如组队组件分为队伍管理、组队挑战、组队切磋和共享展示工具。
- `game/features/<domain>/service.py` 只在一个业务对象确实形成单一事务边界时保留。出现多个互不共享状态的流程，应拆成 `party.py`、`battle.py` 之类的 ASCII 模块。
- `game/content/catalog/package.py` 只做最终组装；内容定义按领域归档，世界皮肤与真实世界定义分开。
- 不为了缩短文件制造 `runtime`、`service`、`content` 的重复目录，也不保留只为内部路径兼容的空壳模块。

## 事务规则

- 一个用户动作涉及多个聚合时，必须由所属 feature 打开一个工作单元并一次提交。
- 幂等记录、快照、战报、奖励和 outbox 由同一事务承接，任何一步失败都不能留下半份业务结果。
- 只读查询可以有自己的工作单元，但不能为了读取同一动作所需的状态而在事务内外来回切换。
- 规则层只返回事实和事件；展示层决定如何把事实变成 QQ 文本、按钮或网页链接。

## 优化节奏

每批只做一个结构主题：先补边界测试，再移动代码，再跑全量测试。命令名称、玩家可见文案、数据库结构和稳定 ID 不因整理而改变。大型文件按语义拆分，小型重复只在重复表达同一边界时抽象。

## 当前顺序

已完成：

- 社会请求支持复用调用方工作单元，组队切磋不再复制快照、幂等记录和 outbox 写入。
- 单人切磋的请求接受与战报写入合并为一个事务，并以故障注入测试证明失败时完整回滚。
- 组队命令按队伍管理、组队挑战和组队切磋拆分，中文组件入口和所有玩家行为保持不变。
- `build_game_services` 已分为正式内容装配、通用玩法底座和业务 feature 装配三个阶段，仍只有一个组合根。
- 命令层统一使用 `command_helpers` 的逻辑时间与当前角色取值，并由回复层承接统一异常记录。
- 探险批次已按一致快照、战斗模拟、奖励结算、资源写入和事实观察分阶段；战报投影独立且整批仍只提交一次。

后续按收益排序：

1. 把重复的命令层时间、角色取值和错误回复整理为共享工具，但不让共享工具反向依赖具体业务。
2. 把 `game/app.py` 的装配段按阶段提取为私有构造函数；仍由 `build_game_services` 统一调用，不增加第二个组合根。
3. 继续审计跨界灾厄、组队挑战和经济购买的长事务，优先消除事务内外重复读取。
4. 按用户流程拆分物品、伙伴等超长命令文件，不按行数机械切割。
