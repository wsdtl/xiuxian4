# 通信驱动器接入模板

公共适配器层只定义跨协议能力：命令注册、消息上下文、发送意图和当前回复目标。具体协议字段必须留在各自驱动器包里。

## 目录结构

```text
launch/adapter/<driver>/
  __init__.py   # 导出驱动器
  message.py    # HTTP webhook / polling / socket 入口
  event.py      # 原始 payload -> 驱动器内部事件
  handler.py    # 队列、去重、命令匹配、Depends 注入
  manager.py    # 发送队列和真实协议 API 调用
  render.py     # 公共 message.Message -> 驱动器原生协议
  target.py     # 协议私有发送目标
  depends.py    # 驱动器私有 Depends
```

QQ 是当前外部驱动器实现，本地驱动器用于进程内测试与巡检。

## 公共回调参数

驱动器只能把这些跨协议字段注入业务函数：

```python
client_id
message
manager
cmd
raw_message
match
message_context
reply_target
adapter_capabilities
```

协议私有字段必须通过本驱动器的 `depends.py` 显式声明：

```python
from launch.adapter import Depends
from launch.adapter.mydriver.depends import current_mydriver_event


async def handler(driver_event=Depends(current_mydriver_event)):
    ...
```

组件 Depends 不能在内部嵌套驱动器私有 Depends，协议依赖必须出现在命令入口签名上。

## 发送边界

正式业务只发送根级 `message/` 定义的 `DocumentMessage` 或 `ImageMessage`。新驱动器必须实现公共消息到本协议的翻译，不能要求业务组件增加平台判断。

普通回复从当前 `MessageContext.reply_target` 取得目标。脱离入站消息发送时，调用方必须显式提供 `ReplyTarget`：

```python
await manager.send(
    SendRequest(
        message=message,
        target=driver_target,
    ),
    client_id,
)
```

框架不保存“最近一次消息”的发送路线，也不根据历史消息猜测目标。未来业务需要通知、订阅或重试时，由独立投递域持久化地址和策略，再构造明确目标交给驱动器。

## 注册到运行时

驱动器完成后，在 `launch/adapter/registry.py` 中增加 `AdapterSpec`。外部驱动器是否与现有驱动器同时消费业务事件必须显式决定；本地驱动器不暴露 HTTP 入口。

## 测试清单

- 原始事件能规整成公共消息上下文。
- 协议私有字段只能通过驱动器 Depends 获取。
- 组件 Depends 不能嵌套协议私有 Depends。
- 当前消息回复能选择正确 manager 和 ReplyTarget。
- 公共 Message 能渲染为本驱动器协议，原生 payload 不能从公共 manager 绕过。
- 显式目标发送不依赖入站上下文。
- 签名、事件去重、队列上限和关闭清理都有独立测试。
