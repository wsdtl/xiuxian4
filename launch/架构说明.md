# 消息框架架构

`launch` 是应用与通信底座，不属于任何业务域。它可以装载业务，但不能导入未来的修仙包。

## 入站链路

```text
平台 HTTP 事件 / 本地 dispatch
  -> 驱动器 message 或 event
  -> 驱动器 handler 解析、去重、排队和命令匹配
  -> MessageContext + MessageIdentity + ReplyTarget
  -> 命令守卫
  -> Depends 参数解析
  -> 业务回调
```

QQ 原始 payload 只能留在 `launch.adapter.qq` 内。驱动器把账号识别需要的已验证字段统一规整为
`MessageIdentity`，业务命令因此只注册一次。业务确实需要身份之外的协议字段时，才在命令入口使用
`launch.adapter.qq.depends` 显式声明；组件自己的 Depends 不得嵌套协议私有 Depends。

公共 `MessageContext.sender_name` 保存平台事件实际携带的发送者名称。该字段允许为空；驱动器不能用
OpenID、客户端 ID 或群 ID 伪造名称。

`MessageIdentity` 保存本次事件的主身份、同事件证明的别名、平台租户和稳定事件 ID。它只表达通信
事实，不创建游戏账号；具体游戏组合层负责把它转换成自己的账号凭据。

## 回复链路

```text
message.Message / SendRequest
  -> launch.adapter.manager
  -> 当前 MessageContext 或显式 ReplyTarget
  -> 对应驱动器 manager
  -> 驱动器消息渲染器
  -> 协议 payload
  -> 平台发送 API / 本地回复收集器
```

业务层不判断当前是 QQ 还是本地驱动器。业务输出统一使用根级 `message/` 的 `DocumentMessage` 或 `ImageMessage`；标题、栏目、正文、附加区、图标和动作都先表达为语义对象。QQ 渲染器负责 Markdown 与 keyboard，本地渲染器负责可断言结果。

公共 `AdapterReplyManager` 不接受平台原生 dict。`launch.adapter.qq.payload` 只属于 QQ 驱动器内部和协议测试，不能成为业务输出协议。

## 脱离上下文发送

框架不记录最近入站目标，也不提供按 `client_id` 猜测发送地址的路由表。普通回复使用当前 `ReplyTarget`；脱离入站上下文发送时，调用方必须提供明确目标。

未来定时通知、订阅和重试由独立投递域负责持久化通信地址与策略，消息框架只执行已经确定目标的发送请求。

## 生命周期

应用启动顺序：

1. 获取单实例锁。
2. 挂载静态资源和驱动器 HTTP 入口。
3. 启动驱动器队列与索引。
4. 启动同步、异步调度器并安装任务。
5. 按优先级执行业务启动回调。

关闭时先执行业务关闭回调，再关闭驱动器和调度器，最后释放单实例锁。

## 扩展约束

- 新驱动器实现自己的 event、handler、manager、target 和私有 depends。
- 新驱动器只通过 `adapter.registry` 接入，不修改业务组件。
- 公共上下文字段必须跨协议成立，不能为了某个平台增加私有字段。
- 业务消息对象与平台 OpenAPI payload 必须分层，不能在业务组件里直接拼协议请求。
- 新驱动器必须实现公共 `message.Message` 的翻译，不得要求业务新增平台分支。
- 框架测试必须覆盖入站规整、依赖边界、回复选择、主动路线和生命周期。
