# 游戏核心边界说明

## 核心版本

```text
game-core.v11
```

`game/core/` 只收纳不依赖命令和通信平台的游戏地基：

```text
game/core/
├─ gameplay/       纯规则、内容目录和规则引擎
├─ account/        平台中立的账号与身份归属
├─ persistence/    数据库、快照、联合事务和 Outbox 适配器
└─ 核心边界说明.md
```

## 三层职责

`gameplay/` 只计算规则结果。它不能读取数据库、机器时间、QQ 事件或消息上下文，也不能导入
`persistence`、`launch`、`message` 和 `组件测试`。

`account/` 只把平台身份映射为稳定内部账号。它不知道 QQ 原生协议、Gameplay 状态和数据库实现，
同样不能导入其他核心层。

`persistence/` 是核心内部唯一允许认识领域快照和 SQLite 的适配层。它可以依赖 `gameplay` 和
`account`，但不能依赖 `launch`、`message`、`组件测试` 或具体玩法命令。

## 核心之外

```text
launch/       应用生命周期和通信驱动器
message/      跨驱动消息协议
auto/         框架自动初始化模块
game/cmd/     命令注册、组件业务与 HTTP 接口
game/content/ 具体游戏的稳定名录与世界皮肤
game/rules/   跨组件复用的具体游戏规则
game/features/ 依赖公共核心、供命令组件调用的正式玩法层
game/app.py   具体游戏服务组装与启动生命周期
组件测试/    可删除的通信协议联调组件
static/       网页资源
design/       游戏设计文档
test/         测试与架构巡检
```

这些目录可以在自己的职责内调用核心，但核心不能反向调用它们。QQ 文案、按钮、命令参数、HTTP
路由和定时 worker 启动方式都不属于核心。

行动领域只维护行动槽、冻结快照和生命周期。未来具体玩法负责解释行动结果，资产变化仍由奖励
结算提交，周期系统只提供时间窗口；三者不能互相复制职责。

## 导入规则

外部代码必须使用完整产品命名空间：

```python
from game.core.gameplay.inventory import InventoryEngine
from game.core.account import AccountIdentityEngine
from game.core.persistence import SqliteDatabase
```

禁止重新建立顶层 `gameplay`、`account`、`persistence` 兼容包。核心内部优先使用相对导入；根
`game.core.__init__` 只公开整体版本和层清单，不聚合所有领域类型。

`game-core.v11` 增加可信抽取池、批量抽取、抽取签绑定和抽取库存原子执行器；随机与保底继续复用
掉落底座，具体签、奖项、概率、命令和来源不进入核心。
`game-core.v10` 为伤害事实增加结构化 `is_proc` 标记，内容触发器可以区分原始攻击和追加伤害，
从地基上阻断装备、套装与被动之间的递归触发。
`game-core.v9` 增加确定性随机 Effect 分支、按层数求值、当前冷却调整、纯标记状态持久化、内容包
Ability 目标规则和批量物品平衡审计；这些能力不依赖任何具体武器名称或世界皮肤。
`game-core.v8` 为库存补齐按单位计量的空间约束和永久物品编号，把装备套装身份移入实例，并以底座族
替代流派定义；生成型武器在启动期校验唯一且不可复用的核心特色。
`game-core.v7` 为世界皮肤条目增加可选紧凑名称，并公开协议中立的字符显示宽度计算；具体角色名字符
范围和境界区间仍由核心之外的正式游戏规则决定。
`game-core.v6` 为世界皮肤补齐自身名称、说明和图标，为角色永久状态补齐玩家名称，并把皮肤身份
纳入内容指纹。
`game-core.v5` 在 `v4` 的队伍地基上增加数据库写入口、状态所有权、动态字段和变更证据门禁。
`game-core.v4` 在 `v3` 的物品化地基上补齐持久临时队伍、成员权限、准备与站位、社会邀请联合
事务及战斗阵营投影。具体组队收益、匹配、玩法准入、命令和展示仍然不进入核心。

## 验证

```powershell
.venv\Scripts\python.exe -X utf8 -B test\core_architecture_test.py
.venv\Scripts\python.exe -X utf8 -B test\core_governance_test.py
```

架构测试使用 Python AST 检查导入关系，并验证旧顶层包不存在、核心根入口保持精简。治理测试继续
检查数据库写入口、跨领域状态、动态字段和底座证据。两者都会拒绝具体组件反向塑造真正核心。
