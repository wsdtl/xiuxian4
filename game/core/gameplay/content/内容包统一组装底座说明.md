# 内容包统一组装底座说明

## 封板版本

```text
content.foundation.v2
```

代码可通过 `gameplay.content.CONTENT_FOUNDATION_VERSION` 读取版本。

## 一句话理解

以后新增世界、资料片、赛季规则或成套玩法时，业务只提交若干版本化内容包：

```text
核心规则包
战斗机制扩展包
区域与玩法包
世界皮肤包
```

`ContentAssembler` 负责按依赖顺序统一登记、检查所有跨目录引用、构造完整运行引擎并一次冻结。
业务组件不能再分别创建自己的物品、角色、技能或武器目录。

## 内容包清单

每个 `ContentPackage` 必须拥有 `ContentPackageManifest`：

```python
ContentPackageManifest(
    id="content.adventure",
    version=ContentVersion(1, 2, 0),
    dependencies=(
        PackageRequirement("content.core", ContentVersion(1)),
    ),
)
```

运行期对同一个包 ID 只能选择一个版本。历史版本可以存在于代码或发布物中，但不能同时进入同一个
`ContentRuntime`。

依赖版本使用：

```text
minimum_version      包含下界
maximum_exclusive    不包含上界
```

缺包、版本不满足、自依赖、重复依赖和依赖环都会在启动组装时失败。

## 确定性装配顺序

装配器先进行拓扑排序：

1. 依赖包永远先于使用方。
2. 同一拓扑层按包 ID 排序。
3. 调用方传入包的顺序不影响结果。

因此内容指纹、冲突报错和运行目录不会随文件扫描顺序或操作系统变化。

## 类型化内容范围

当前内容包可以声明：

```text
物品组件类型
货币与品质
属性与资源
角色模板、成长轨道、永久特征
物品、武器、装备流派与装备
属性价值、机制价值与协同价值
随机属性、生成策略与装备套装
临时队伍类型与容量
战斗公共配置
伤害类型、控制、伤害干预器、目标约束
Effect、Ability、Trigger
日、周、月、固定间隔、赛季与活动周期
世界皮肤及其历史版本
```

武器品质档案、装备品质档案、Effect 操作、Ability 引用等仍使用各领域已经定义好的类型，不在内容
层复制第二套模型。

## 受控扩展点

内容包可以登记：

- 新数值表达式及其求值器和引用校验器。
- 新规则条件及其执行器和引用校验器。
- 新 Effect 原子操作及其执行器和引用校验器。
- 新伤害干预处理器。
- 新目标选择器。
- 新物品组件类型。
- 新周期调度类型。

这些扩展都有明确类型或稳定 ID，重复登记会失败。内容包不能获得任意 `install(context)` 回调，不能
借统一组装之名直接修改已经登记的定义或提前冻结目录。

## 隐藏依赖审计

仅仅在清单里写依赖还不够。装配器会递归检查包内所有 dataclass、集合和映射：

- 如果稳定内容 ID 属于另一个包，当前包必须直接或传递依赖其所有者。
- 如果使用了另一个包登记的数值、条件或 Effect 操作类型，也必须声明依赖。
- 世界皮肤引用其他包的展示内容，同样必须依赖对应内容包。

这样一个包不会因为“这次碰巧和另一个包一起加载”而运行，单独禁用或替换包时也不会留下隐式引用。

## 全局稳定 ID 所有权

每个已登记稳定 ID 只能有一个 `ContentOwner(package_id, category)`。冲突信息会同时指出两个包和
内容类别，例如物品、货币、技能、装备或实现扩展。

武器定义和绑定物品使用不同命名空间：

```text
item.weapon.training_blade
weapon.training_blade
```

玩家可见名称、别名和世界皮肤名称永远不能成为所有权键。

## 分阶段组装

固定阶段如下：

1. 登记内置和扩展处理器、物品组件类型。
2. 登记货币、品质、属性、资源和基础规则定义。
3. 登记角色、物品、战斗定义、Effect、Ability、Trigger 和周期。
4. 冻结价值与物品化目录，并严格校验每个随机属性档位的边界值。
5. 在共享物品、品质与物品化目录上登记武器、装备流派、套装和装备。
6. 审计包依赖、稳定 ID 所有权和展示集合。
7. 选择战斗配置并构造所有规则引擎。
8. 执行各领域已有的跨引用校验。
9. 登记并校验全部世界皮肤。
10. 冻结所有目录并生成内容指纹。

内容包的输入顺序不能改变这些阶段。

## 战斗配置与实际运行引擎

`CombatProfileDefinition` 统一提供：

```text
CombatStats
RecoveryStats
ControlStats
DamageRules
```

只有一个配置时自动选中；存在多个配置时必须显式传入 `active_combat_profile_id`。组装结果不只是定义
集合，而是已经完成引用校验的：

```text
DamageEngine
RecoveryEngine
ControlEngine
EffectEngine
AbilityEngine
TriggerEngine
TargetSelectorRegistry
```

标准伤害、治疗、护盾、控制和时间线 Effect 操作会自动接入同一套引擎。自定义操作也走同一注册表，
不会创建旁路战斗系统。

## 世界皮肤

每个内容包显式声明需要玩家展示的 `display_content_ids`。所有激活皮肤版本必须完整覆盖最终并集，
并且不能包含未知条目。

每个 `SkinPack` 自身也必须声明稳定 ID、版本、玩家可见名称，并可提供说明和图标。不同皮肤不能
使用相同名称；这些身份字段会进入内容指纹。

世界皮肤数量没有上限，也不存在主世界、备用世界或固定两个槽位：

```text
skin.cultivation@1
skin.cultivation@2
skin.magic@1
skin.science_fiction@1
```

新增玩法包增加展示内容后，所有准备继续启用的世界皮肤都必须随发布补齐，否则启动失败。

## 运行期冻结

`ContentRuntime` 返回前，以下目录全部冻结：

```text
货币、品质、物品与组件
角色、武器、装备
临时队伍类型
属性、资源、战斗配置
伤害、控制、干预、目标约束
Effect、Ability、Trigger
周期定义和调度处理器
目标选择器和世界皮肤
```

运行期间不能偷偷注册技能、修改掉落引用或覆盖定义。内容变化只能通过新的内容包版本和重新组装完成。

## 内容指纹

`ContentAssemblyReport.content_fingerprint` 是 64 位 SHA-256，输入包括：

```text
底座版本
确定后的包顺序与完整版本
所有类型化内容值
受控扩展的类型和处理器身份
当前战斗配置
世界皮肤内容
周期定义及自定义调度处理器身份
```

相同包集合无论传入顺序如何都会得到相同指纹。任何定义值、包版本、皮肤或激活战斗配置变化都会
改变指纹。

处理器函数使用模块名和限定名作为稳定身份。数值求值器、条件、Effect 操作、干预器或目标选择器
的函数实现发生变化时，所属内容包必须提升版本；装配器不会读取 Python 源码文本推测代码是否改变。

## 数据库激活闭环

`persistence.ContentActivationStore` 把激活包集合、战斗配置和内容指纹保存到数据库：

- 第一次启动可以初始化指定激活槽。
- 后续普通启动只能验证完全一致的内容。
- 指纹不同默认抛出 `ContentActivationMismatch`。
- 发布新内容必须显式调用 `replace()`，同时提供旧 revision 和旧指纹。
- 内容切换使用 CAS，两个发布流程不能互相覆盖。

默认槽为 `content.active`，也可以为不同世界或规则分片使用不同槽。内容层不导入数据库；适配器位于
`game/core/persistence/`。

## 使用示例

```python
from game.core.gameplay.content import ContentAssembler
from game.core.persistence import ContentActivationStore

runtime = ContentAssembler(
    active_combat_profile_id="combat_profile.standard",
).assemble((core_package, adventure_package, skin_package))

activation = ContentActivationStore(database).verify_or_initialize(
    runtime.report,
    logical_time=startup_time,
)
```

业务组件只注入 `ContentRuntime` 中的目录和引擎，不能再次组装一套私有内容。

## 不属于本底座

- 具体物品、技能、武器、装备或世界皮肤内容。
- 掉落概率、任务条件、商店价格和活动奖励。
- 玩家资产、角色状态和数据库快照。
- QQ 文案、按钮和网页展示布局。
- 日、周、赛季和活动的具体结算业务。
- 历史数据迁移和旧内容兼容。

## 重新审核条件

普通新增包、定义、世界皮肤或受控扩展不需要重开底座。只有以下情况需要更新封板版本：

- 内容无法归入现有类型化清单或受控扩展协议。
- 一个运行期必须同时激活同一包 ID 的多个规则版本。
- 现有拓扑和隐藏依赖审计无法表达新的加载关系。
- 内容指纹不能覆盖必须审计的运行语义。
- 运行期必须允许修改已经冻结的定义。
