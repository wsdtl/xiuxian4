# 角色与成长底座说明

## 封板版本

```text
character.foundation.v4
```

代码可通过 `gameplay.character.CHARACTER_FOUNDATION_VERSION` 读取版本。

`character.foundation.v4` 增加协议中立的字符显示宽度计算，供聊天类具体游戏限制单行名称；核心角色
模型仍只做通用名称规范化，不接受具体世界的字符范围反向污染。

`character.foundation.v3` 在规范化名称和账号角色目录之外，明确最大血气成长的资源语义：成长里程碑
提高最大血气时，当前血气同步增加相同固定值，保持缺失血气不变，而不是借升级直接回满。

`CharacterRosterState` 以账号 ID 为聚合键保存角色 ID 目录。持久化服务可以在同一个事务内提交目录
和角色快照，并由具体游戏传入账号角色数量上限。

## 核心边界

角色档案通过 `account_id` 归属于稳定内部账号，并保存五项核心值、当前血气与灵力、成长轨道和永久特征。它不保存最终战斗面板，
也不为每一种装备词条增加数据库字段。

五项核心属性固定为：

```text
health.maximum          最大血气
spirit.maximum          最大灵力
combat.attack           基础攻击
combat.defense.physical 基础防御
combat.speed            行动速度
```

标准边界：

- 最大血气最低为 1。
- 最大灵力、基础攻击和速度最低为 0。
- 防御允许为负，由战斗规则把负防折算为增伤。
- 五项核心值不设置统一硬上限，内容数值和场景规则负责自己的上限。

角色持久资源只有：

```text
health.current
spirit.current
```

护盾、连击层数、控制状态和临时战斗资源不写入角色永久档案。

## 不存在固定的十五项调节值

暴击、闪避、穿透、格挡和控制抵抗不是角色固定字段。装备、体质、宗门或场景确实需要时，通过
`CharacterContribution` 提交对应稳定属性 ID：

```text
武器实例 -> combat.critical.chance +0.20
鞋子实例 -> combat.evasion +0.15
宗门增益 -> healing.outgoing +0.05
```

属性只需先登记到当前 `AttributeResolver`，角色投影器即可接受。新增普通战斗属性不需要修改
`CharacterState`、角色表或角色投影器。

吸血、反伤、追击、复活等行为不应伪装成角色数字，应通过贡献授予现有的 Ability、Trigger、
伤害干预器或目标约束。

## 角色模板

`CharacterTemplateDefinition` 只决定新角色的：

- 五项核心初始值。
- 初始成长轨道。
- 初始永久特征。
- 身份性质标签。

玩家可见名称、境界文字和图标由世界皮肤处理，不能成为模板业务键。

具体游戏可以按字符显示宽度限制角色名。修仙4 当前限制为 12 个显示宽度，只允许中文、英文字母和
数字；该产品规则位于 `game/rules/character/identity.py`，不属于核心角色模型。

## 成长轨道

等级、武器掌握和其他真正拥有独立经验的长期成长使用 `ProgressionDefinition`。具体游戏如果仅把境界
作为人物等级的展示分段，应由等级实时推导，不能再保存第二条境界经验或重复状态。经验需求采用明确数值表：

```text
(100, 200, 360)
```

表示 1 升 2 需要 100，2 升 3 需要 200，3 升 4 需要 360。引擎不隐藏计算公式，一次获得大量
经验可以连续跨越多个等级。

里程碑只能：

- 修改五项核心值。
- 解锁一个已经登记的永久特征。

暴击等非核心属性需要通过永久特征贡献，不能直接写进角色状态。

## 永久特征与外部贡献

`CharacterFeatureDefinition` 表达体质、血脉、天赋等角色永久拥有的规则来源。

`CharacterContribution` 是装备、宗门、场景和活动使用的公共接口。一份贡献可以提供：

- 任意已登记属性的固定值或比例修改。
- 标签。
- Ability。
- Trigger。
- 伤害干预器。
- 目标约束。

每份贡献携带 `source_kind` 和 `source_id`。属性明细因此能够指出数值来自哪件装备、哪个永久
特征或哪个场景，而不是只剩一个无法解释的最终总数。

## RuleEntity 投影

`CharacterProjector` 是角色进入战斗和其他规则场景的唯一入口：

```text
五项核心值
+ 永久特征贡献
+ 装备贡献
+ 宗门贡献
+ 场景贡献
= RuleEntity
```

投影器只组合现有规则积木，不参与伤害公式。未登记属性和未知 Ability、Trigger、干预器或目标
约束会在投影组装时失败。

角色状态允许保存高于裸装核心上限的当前血气或灵力，因为上一场结算可能包含装备加成。投影时
会先汇总全部最大值贡献，再把当前资源夹紧到最终上限；降低装备上限不会带入超额当前血气。

创建新角色后使用 `CharacterProjector.initialize_new_character(...)`，会先计算初始永久特征，再把
当前血气与灵力填充到最终上限，避免提高上限的初始体质让新角色以非满状态出生。

## 原子成长事务

`CharacterEngine` 支持：

- 增加成长经验并连续升级。
- 解锁新的成长轨道。
- 应用成长里程碑。
- 永久修改核心值。
- 解锁永久特征。
- 修改当前血气或灵力。
- 将角色标记为退隐。

事务携带 `expected_revision`，拒绝用旧角色快照覆盖新状态。任意一步失败时，角色原快照和随机
游标都不改变。

当前结构化事实包括：

```text
character.experience.gained
character.progression.unlocked
character.progression.advanced
character.milestone.applied
character.core_attribute.changed
character.feature.unlocked
character.resource.changed
character.retired
```

## 不属于本底座

- QQ、微信或其他平台身份绑定。
- 玩家可见名字、称号和境界文案。
- 纳戒物品和装备实例。
- 货币余额和税务。
- 宗门、城池、任务和活动关系。
- 宝石、孔位和其他未启用的装备附加玩法。
- 战斗伤害和回合结算。
- 数据库表、缓存和分布式锁。

账号系统决定一个账号可以创建几个角色；武器与装备系统负责产生装配贡献；战斗结算负责明确提交需要
写回的当前血气和灵力。它们不能直接修改角色对象内部映射。

## 重新审核条件

以下扩展不需要重开角色底座：

- 新增成长轨道。
- 新增永久特征。
- 新增角色模板。
- 新增由装备或场景提供的属性。
- 新增 Ability、Trigger 和特效组合。

只有需要改变五项核心值、持久资源边界、成长事务原子性或贡献来源协议时，才更新封板版本。


任意属性贡献、来源明细和最终资源夹紧。
