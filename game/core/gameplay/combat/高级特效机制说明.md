# 高级特效机制说明

## 设计原则

高级特效不按“吸血类、毒类、某把武器类”写进战斗主流程。内容只组合通用数值
公式、Effect 操作、伤害干预器、Trigger 和时间线指令。真正出现全新规则阶段时
才注册新处理器。

## 复合数值公式

当前公式可以读取：

- 来源或目标属性。
- Effect 参数和事件数值。
- 来源或目标的当前资源。
- 缺失资源、当前比例和缺失比例。

公式可以继续组合：

```text
SumMagnitude       相加
ProductMagnitude   相乘
MinimumMagnitude   取最小
MaximumMagnitude   取最大
ClampMagnitude     限制区间
RatioMagnitude     相除并处理零分母
PowerMagnitude     幂曲线
```

例如“攻击力 + 目标缺失血气的 20%，最高不超过攻击力三倍”不需要新增 Python
伤害公式。所有嵌套属性和资源引用会在启动阶段校验。

## 状态生命周期

Effect 支持：

```text
DispelEffects          按定义、语义标签和来源净化完整状态
ConsumeEffectStacks    消耗指定层数
ModifyEffectDuration   延长或缩短剩余回合
ModifyCooldown         增减、设置或清零 Ability 冷却
```

叠层 Effect 保存每一层贡献的属性修改数量。移除一层时会同步撤销该层属性，不会
只改显示层数却保留旧增益。

## 资源互动

`TransferResource` 从目标实际可用资源中抽取，并按效率交给来源。双方分别经过
资源上下限，事件会记录：

- 请求抽取量。
- 目标实际损失量。
- 来源实际收到量。
- 因来源已满而溢出的量。

因此吸取灵力、燃烧能量、生命交换等机制不需要直接修改两个实体。

## 正式治疗和护盾

`RecoveryEngine` 统一结算：

- 施术者治疗加成。
- 目标受疗加成或禁疗。
- 实际恢复。
- 过量治疗。
- 治疗前后血气。

`Heal` 产生 `combat.healing.resolved`，Trigger 可以据此制作过量治疗转盾、治疗
后追击或受疗反制。

`GrantShield` 可以设置相对目标最大血气的护盾上限，并产生
`combat.shield.granted`。

## 伤害干预器

持续 Effect 可以通过 `GrantInterceptor` 授予伤害干预器。固定阶段为：

```text
RAW
→ AFTER_CRITICAL
→ AFTER_DEFENSE
→ AFTER_RATES
→ AFTER_BLOCK
→ BEFORE_SHIELD
```

当前内置处理器：

```text
interceptor.multiply                    伤害倍率
interceptor.flat                        固定增减
interceptor.cap                         伤害封顶
interceptor.immunity                    伤害免疫
interceptor.bypass_shield               穿透护盾
interceptor.death_guard                 致命伤保留最低血气
interceptor.convert                     转换伤害类型
interceptor.redirect_to_grant_source    向状态授予者分摊伤害
```

每次干预产生 `combat.damage.intercepted`，包含干预前后数值、伤害类型、阶段、
拥有者和授予来源。

伤害分摊会产生以保护者为目标的 `combat.damage.redirected`。保护者的 Trigger
再通过 `DealDamage` 承受分摊伤害，因此分摊部分仍会正常经过其防御、护盾、
免疫、反伤和击败规则，递归链由既有深度上限保护。

新增真正不同的伤害规则时，注册 `InterceptorHandler` 即可，不修改
`DamageEngine`。

## 控制协议

`ControlEngine` 统一计算：

```text
基础命中率
+ 来源控制命中
- 目标控制抵抗
→ 确定性随机判定
→ 基础持续时间 × (1 - 韧性)
→ 至少一回合
```

`ApplyControl` 成功后授予控制语义标签并覆盖 Effect 持续时间；失败仍产生
`combat.control.resolved`，日志可以准确展示抵抗。禁疗、免控等绝对规则仍可由
Effect 的目标标签边界处理。控制抵抗以及战斗中未满足标签或动态条件的 Effect 还会产生
`effect.application.rejected`，因此战报不会再把失败施加显示成零层效果。

## 目标约束

持续 Effect 可以通过 `GrantTargetConstraint` 授予服务端强制目标规则。当前通用
约束包括：

```text
FORCE_GRANT_SOURCE   单体行动必须选择该状态的授予者，例如嘲讽
UNTARGETABLE         对指定阵营关系不可选中，例如隐身或相位状态
```

目标约束在选择器完成后、Ability 执行前统一检查。提交其他目标只会得到
`target.no_valid_target`，不会扣资源或推进回合。特殊侦测技能可以在服务端
`BattleAbilityTargeting` 中声明绕过约束，玩家请求本身不能绕过。

## 行动指令

Effect 可以向编排层提交：

```text
RequestExtraTurn    请求额外行动
RequestTurnDelay    延后尚未行动的目标
RequestInterrupt    中断当前 Ability 的后续 Effect 段
```

额外行动有单次行动上限，整场战斗还有最大总行动数和最大轮数，避免无限刷新。
打断只停止尚未执行的段数，已经产生的伤害和资源变化不会回滚。

## 动态参与者

`BattleEngine.join(...)` 允许召唤物、分身或援军中途加入。加入者从下一轮开始
行动，避免插入当前行动造成顺序歧义。

`BattleEngine.withdraw(...)` 支持逃跑、退场和替换。退出者：

- 不再成为目标。
- 不再拥有可触发被动。
- 不再进入行动顺序。
- 仍作为只读历史实体保留。

保留历史实体很重要：施毒者退出后，其已施加的中毒仍能继续结算，并保持正确
伤害与击败归属。

`ActiveEffect` 同时保存施加时参数。周期 Trigger 会把这些快照参数传给后续
Effect，因此持续伤害可以使用施加时已经确定的强度，不必在每次跳伤时重新读取
已退出施法者的临时面板。

Trigger 还会注入 `effect.stacks`，叠层中毒可以直接用
`ParameterMagnitude("effect.stacks")` 参与周期伤害公式。

## 可以直接组合的复杂效果

当前不修改主流程即可定义：

- 残血越低伤害越高，并设置固定上限。
- 三层印记引爆后消耗两层。
- 净化一个负面状态并缩短剩余状态持续时间。
- 吸取目标灵力，溢出部分销毁。
- 过量治疗转换为护盾。
- 致命伤保留一点血气。
- 受到特定伤害时转换为真实伤害或免疫。
- 守护者承担一半伤害。
- 击败后清零冷却并获得额外行动。
- 受击后打断攻击者尚未执行的连击。
- 召唤物加入、行动并在规则要求时退出。
- `combat.target.defeated` 后通过 `Heal` 实现复活；血气从最低值恢复为正数时同步产生
  `combat.target.revived`。复起仍使用现有触发链和行动顺序，不另设结算阶段。

这些只是内容定义示例，不属于内核写死的流派。
