# 战斗内核说明

## 解决什么问题

技能、武器和体质不能各自直接扣除血气。它们只提交 `DamageRequest`，由
`DamageEngine` 使用同一条流水线结算。这样吸血、反伤、日志和击败判断读取
的都是同一份实际伤害，不会把过量伤害当成真实伤害。

```text
原始伤害
→ 命中
→ 暴击
→ 防御与穿透
→ 全局及伤害类型增减伤
→ 格挡
→ 固定值与最大血气比例限制
→ 护盾
→ 实际血气伤害
→ 击败
```

## DamageTypeDefinition

伤害类型是可注册定义，不使用封闭枚举。每种伤害可以分别指定：

- 防御属性。
- 固定穿透与比例穿透属性。
- 该类型的造成伤害和承受伤害属性。
- 是否忽略防御。
- 语义标签。

因此物理、术法、真实伤害只是内容定义。以后增加神魂、毒素或场景专属伤害
不需要修改 `DamageEngine`。

## 负防御

防御大于等于零时使用递减收益曲线：

```text
倍率 = K / (K + 防御)
```

防御小于零时平滑转为增伤：

```text
倍率 = 2 - K / (K - 防御)
```

两段公式在零点连续。负防御越低，伤害越接近两倍，但不会出现除零奇点或无限
膨胀。`K` 由 `DamageRules.defense_constant` 配置。

## 固定值和比例边界

每次 `DamageRequest` 可以同时指定：

- 最低伤害。
- 最高固定伤害。
- 最高目标最大血气比例伤害。

多个上限同时存在时取更严格者。技能可以表达百分比伤害，但不能绕过场景规定
的固定值边界。

## 护盾与实际伤害

护盾默认先于血气吸收伤害，也可以由请求显式绕过。结算分别记录：

- 护盾实际承受值。
- 血气实际承受值。
- 过量伤害。
- 结算前后血气和护盾。

吸血应读取 `health_damage`，需要按护盾伤害生效的机制应读取 `shield_damage`，
两者不能混用。

## Effect 接入

`DealDamage` 是 Effect 进入正式伤害流水线的原子操作。普通攻击、技能、反伤和
周期伤害都应复用它，不再使用负数 `ChangeResource` 直接扣除血气。

```python
DealDamage(
    id="operation.slash",
    damage_type="damage.physical",
    magnitude=AttributeMagnitude("combat.attack"),
)
```

一次 Effect 需要多段攻击时，应编排多个 Effect 引用，使每一段独立结算护盾、
击败和触发链，不要在一个 Effect 中并列多个 `DealDamage`。

每段 Effect 产生的 Trigger 会在下一段开始前完成。第一段触发的护盾、反击、
增益或减益会真实影响后续段数，不会等整招结束后才补算。

## 结构化战斗事实

当前产生以下稳定事实：

```text
combat.attack.missed
combat.attack.hit
combat.attack.critical
combat.attack.blocked
combat.shield.damaged
combat.shield.broken
combat.damage.dealt
combat.damage.prevented
combat.damage.intercepted
combat.damage.redirected
combat.target.defeated
```

吸血监听攻击者产生的 `combat.damage.dealt`，反伤监听目标收到的
`combat.damage.dealt`。零伤害只产生 `combat.damage.prevented`，不会启动伤害
连锁。

概率追击、施毒等效果使用 `TriggerDefinition.chance`。概率由规则上下文的确定性
随机源统一产生，组件不能自行调用系统随机数。

## 当前边界

本层只负责一段伤害。回合、目标、控制、持续伤害时点和胜负已经由同目录的
`BattleEngine` 承接，详见《战斗编排说明》。奖励、掉落、自动战斗 AI 和玩家
输入仍不属于战斗规则内核。
