# 物品与物资底座说明

## 封板版本

```text
inventory.foundation.v6
```

代码可通过 `gameplay.inventory.INVENTORY_FOUNDATION_VERSION` 读取版本。

## 一句话理解

库存域只回答八个问题：

```text
谁拥有什么
物品在哪里
数量是多少
哪项业务正在占用它
这次变化是否完整成功
物品占用了多少受限空间
玩家用哪个永久编号引用它
哪些武器装备受到玩家珍藏保护
```

它不决定物品值多少钱、从哪里掉落、装备后增加多少攻击，也不把玩家可见名称当作业务键。

## 静态定义

`ItemDefinition` 使用稳定英文 ID，并明确资产形态：

- `STACK`：药物、矿石、建设材料等可堆叠物资。
- `INSTANCE`：武器、装备、带随机词条物品等独立实例。

世界皮肤负责名称、图标和描述。物品定义只保存标签、堆叠上限和类型化组件。

独立实例的 `data` 保存武器、装备等与资产同生共灭的类型化状态。铭刻由独立铭刻底座按 `asset_id`
保存，并由持久化联合事务保证它在资产转移、销毁时保持一致；库存域不偷存第二套铭刻规则。

`ItemComponentRegistry` 允许后续系统登记新的组件槽位和数据类型。例如消耗品登记
`item_component.use_ability`，具体游戏的可回收物品登记 `item_component.recycle_value`。未知组件、
错误数据类型和错误引用必须在启动组装时失败，不能等玩家使用时才发现。

## 资产与容器

每份资产拥有全局唯一 `asset_id` 和库存内单调分配的整数编号，并且只能指向一个容器。编号在移动、
穿戴和转移时保持不变；资产销毁或堆叠合并后释放映射，但历史编号永不复用。展示层可以按类别渲染为
`W184`、`E327` 或 `I52`，核心只保存整数。容器同时定义当前位置和当前所有者：

```text
玩家纳戒
装备栏
交易托管
市场寄售
邮件附件
掉落暂存
系统回收
```

物品从一个容器移动到交易托管后，不会在原容器保留第二份副本。容器可以限制资产形态、物品标签、
最大资产条目数和最大空间。空间受限容器要求物品声明 `ItemStorageComponent.unit_space`；堆叠物占用
`unit_space * quantity`，实例物占用 `unit_space`。

具体游戏当前把玩家存储分为纳戒、武器空间和背包：纳戒保存恢复药、特殊消耗品和铭刻之羽，武器空间
保存武器和六类装备，背包保存战利品、任务物资和活动物资。只有背包设置空间上限，穿戴区仍是独立
七格。容器归属不等于业务分类：恢复药与特殊物品虽然同在纳戒并复用堆叠事务，仍由互斥标签和不同
使用组件隔离；铭刻之羽则保持独立实例。

## 来源追踪

物品首次进入系统必须携带 `SourceReceipt`。可堆叠物资内部保存一个或多个
`ProvenanceLot` 来源批次：

```text
灵铁 x30 <- 探险奖励 A
灵铁 x20 <- 首领奖励 B
```

合并后仍保留两个来源批次；拆分和消耗按批次先进先出。这样以后可以追查异常产出、回收指定
来源的资产，也可以解释一批物资是怎样进入系统的。

## 原子事务

`InventoryEngine` 在事务私有副本中依次执行操作，全部成功后才产生新 `InventoryState`。
任意一步失败都会返回 `RuleFailure`，调用方持有的原快照不变，随机游标也会回滚。

当前标准操作：

- 发放堆叠物资和独立实例。
- 向既有堆叠追加带独立来源凭据的数量。
- 消耗堆叠物资和独立实例。
- 移动或跨所有者转移资产。
- 在两个容器之间原子交换资产，供满容器换装等流程使用。
- 拆分与合并物资堆。
- 预约、锁定、托管和释放。
- 珍藏与取消珍藏武器装备。
- 销毁资产。

跨玩家交易应把双方资产移动放入同一个事务。价格和货币转账以后由经济账本参与更外层的
联合工作单元，不能把货币伪装成普通物品。

`transaction_id` 是持久化层建立唯一约束的业务键。当前纯规则底座不保存无限增长的历史事务
集合；接入数据库时必须用唯一索引防止同一事务重复提交。

## 预约、锁定与托管

资产占用使用显式 `AssetReservation`：

- `reserved`：为奖励选择、制作或普通流程预留数量。
- `locked`：禁止其他流程动用，例如待确认操作。
- `escrowed`：资产已经进入交易、决斗或市场托管流程。

没有预约时统一查询为 `available`。预约记录业务类型、业务实例 ID、数量、逻辑创建时间和可选
失效时间。调用方要消耗已预约资产，必须带回对应 `reservation_id`；普通操作只能使用未预约数量。

过期判断只使用 `RuleContext.logical_time`，不读取机器当前时间。成功事务会先释放已过期预约并
产生 `inventory.item.released` 事实。

## 珍藏保护

`InventoryState.protected_asset_ids` 明确保存受保护的武器装备实例，不复用预约状态，也不写入
`ItemInstance.data`。珍藏只限制资产离开当前所有者：销毁、消费、跨所有者移动、跨所有者交换和
`escrowed` 托管都会在库存事务中拒绝；同一所有者下的装配移动和实例数据更新仍然允许。

加入珍藏与取消珍藏都是幂等操作。目标已经处于所需状态时不产生事实，也不递增库存版本。存在业务
预约的资产不能临时加入珍藏，避免已经上架或处于其他流程中的物品被中途改成不可结算状态。

## Ability 联动

标准 `ItemAbilityComponent` 让药物、符箓和主动道具声明一个 Ability。调用
`InventoryAbilityExecutor` 时：

1. 验证具体资产、所有者和物品组件。
2. 在库存私有结果中支付物品成本。
3. 临时授权并执行 Ability。
4. 两边都成功才返回库存状态、角色状态和完整事件。

Ability 失败不会扣物品；库存支付失败也不会接受 Ability 结果。物品只是 Ability 的授权与成本
来源，不需要复制治疗、伤害、控制或触发规则。

### 角色持久资源桥接

`CharacterItemUseEngine` 把具体物品资产、施法者、目标角色和 Ability 组合为一次纯规则结果。它先
使用 `CharacterProjector` 生成规则实体，再执行 `InventoryAbilityExecutor`，最后只把当前血气和
当前精神的实际变化通过 `CharacterEngine` 写回角色快照。

这道桥接刻意拒绝把以下战斗运行状态写入长期角色档案：

```text
临时属性修改
持续 Effect
Ability 冷却
临时标签、能力和触发器
非持久资源
```

因此，同一个 `ItemAbilityComponent` 可以在战斗时间线中产生临时效果，也可以在场外联合事务中
恢复长期资源；场外入口遇到无法持久化的效果会整体失败且不扣物品。自身使用与指定其他角色为
目标使用采用同一个 `CharacterItemUse`，具体玩法只负责目标权限，不能绕过所有权校验。

`consume_quantity=0` 的可重复道具虽然不会扣除资产，也必须通过预约校验。被锁定、预留或托管的
道具不能绕过对应 `reservation_id` 使用；过期预约按传入的逻辑时间忽略。

`ItemUseReceipt` 只记录稳定 ID、消耗数量和每个角色的实际资源变化，不保存物品名称、按钮或回复
文本。`item_use_fingerprint()` 为同一请求生成稳定事务指纹，不把运行期 revision 混入业务身份。

## 不属于本底座

- 货币余额、价格、税收和市场撮合。
- 掉落表、随机权重和保底。
- 装备属性、孔位、宝石、强化和耐久规则。
- 制作配方与建设消耗规则。
- 角色负重、具体容器容量数值和扩容玩法。
- 数据库表、缓存和分布式事务实现。
- QQ 命令与玩家可见文案。

这些系统可以提交库存事务或登记物品组件，不能各自直接改资产数量。

## 重新审核条件

普通新增物品、容器类型和类型化组件不需要重开底座。只有以下情况需要更新封板版本：

- 一份资产必须同时存在于多个位置。
- 现有预约数量模型无法表达新的业务占用。
- 原子事务无法承载新的资产变化。
- 来源批次无法表达必须审计的来源关系。
- 需要改变已发布事件的实际语义。
