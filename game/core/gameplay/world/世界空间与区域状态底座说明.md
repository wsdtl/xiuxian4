# 世界空间与区域状态底座说明

## 封板版本

```text
world.foundation.v4
```

## 真实世界与坐标锚点

`WorldDefinition` 使用 `world_id` 声明真实玩法世界、独立 `space_id`、展示 `skin_id` 和出生锚点。
`MapAnchorDefinition` 保存某一世界地图中的物理坐标，不声明玩法或展示；`WorldLocationBinding` 再以
`(world_id, anchor_id)` 绑定该世界此处的唯一主功能、跨世界展示身份 `display_ref`、具体玩法内容
`content_ref` 和版本。

因此相同功能可以在不同世界拥有完全不同的坐标和布局，但同一世界的一个锚点不能拥有两个主功能。
`WorldRuntimeCatalog.resolve()` 与 `resolve_position()` 是业务解析世界、锚点、位置、功能和具体内容引用的
唯一入口。组件不得再按坐标二次查询探险区域或人物；皮肤不得反向决定位置或玩法规则。

跨世界对应地点通过 `display_ref` 匹配，不复用物理锚点 ID。跃迁可以把同一地点身份映射到目标世界自己的
锚点和坐标；目标世界没有对应身份时才回到出生点。

## 空间与位置

一套游戏可以登记任意数量的世界空间。空间可以是：

- 有整数边界的网格，支持自由坐标和曼哈顿移动成本。
- 由地点与带权连接组成的节点图，支持最短路径移动成本。

地点、空间和连接只使用稳定 ID。名称、地图图片和文字描述由世界皮肤与具体内容包提供。真实世界中的
角色位置使用独立空间坐标，锚点由运行目录反向解析，不把展示地点名写进存档。

`presences_at()`、`presences_within()` 和 `active_reservations_at()` 提供稳定排序的只读位置、范围与
有效预约查询。节点图不可达目标不会混入范围结果，查询过期预约也不会顺手修改世界状态。

## 多存在体

`WorldPresence` 是一个主体在世界中的具体存在。一个角色可以同时拥有本体、行商化身、远征队等
任意数量和类型的存在体，每个存在体分别保存位置和 revision。核心不假设玩家只有一个
`current_location`。

## 稀疏占用

`WorldReservation` 可以预约地点或自由坐标，支持：

- 排他占用，例如山门、地皮或唯一入口。
- 非排他容量占用，例如队伍席位或摊位。
- 数量、所有者和逻辑过期时间。

过期只读取 `RuleContext.logical_time`，读取地图不会顺手清理或改变世界状态。

## 类型化共享计量

世界共享状态只接受预先登记的整数 `WorldMeterDefinition`。每个计量明确最小值、最大值、初始值、
允许的全局/空间/地点作用域、溢出策略和阈值。仙城建设、战备蓄能、区域热度等具体内容可以声明
各自计量，但核心不保存任意 JSON 状态袋。

计量变化产生包含前值、请求值、实际值和跨越阈值的结构化事实。衰减、结算和奖励由周期与游戏
次核心显式提交计量事务。

## 不属于本底座

- 世界名称、地点内容、地形数值和移动消耗公式。
- 导航命令、地图展示、传送道具和玩家权限。
- 探险、宗门、城池、首领和虫洞规则。
- 计量变化应该产生的业务奖励。
