# 时间与周期底座说明

## 封板版本

```text
cycle.foundation.v1
```

代码可通过 `gameplay.cycles.CYCLE_FOUNDATION_VERSION` 读取版本。

## 一句话理解

业务不再自己判断“是不是新的一天”、保存上次执行时间或在启动时补发奖励。内容包声明周期，纯规则
引擎根据显式逻辑时间发现到期窗口，持久化服务把每个窗口变成可防重、可重启、可抢占的工作项。

```text
内容包周期定义
  -> CycleEngine 计算窗口和补偿范围
  -> PersistentCycleService 原子推进游标并创建工作项
  -> worker 使用工作项事务 ID 执行业务结算
```

## 时间边界

所有绝对时间必须是带时区的 `datetime`。规则层和持久化层禁止读取机器当前时间：

```text
禁止 datetime.now()
禁止 datetime.utcnow()
禁止 date.today()
禁止 time.time()
```

调用方必须显式传入 `logical_time` 或 `through`。数据库中的周期时间统一转换成 UTC ISO 字符串，
避免不同时区偏移的字符串排序与真实时间顺序不一致。

日历边界使用指定时区的当地墙上时间。例如上海每日 04:00：

```python
CycleDefinition(
    "cycle.daily_reset",
    CalendarSchedule("Asia/Shanghai", CalendarUnit.DAY, time(4, 0)),
)
```

日历日不是固定 24 小时。夏令时地区的一天可以是 23 或 25 小时，周期引擎按当地日历边界计算。

## 调度类型

内置三类调度：

- `FixedIntervalSchedule`：从锚点开始的固定间隔，可声明短于间隔的有效窗口，从而形成空档。
- `CalendarSchedule`：按时区计算日、周、月；周起始日和当地边界时间可以配置。
- `ExplicitSchedule`：显式声明互不重叠的赛季、节日或活动窗口。

内容包可以通过 `ScheduleHandlerRegistration` 增加新的类型化调度。扩展必须同时提供范围查询、最新
到期窗口查询和当前窗口查询，并接受内容包依赖审计与运行期冻结。

## 窗口与结算延迟

每个 `CycleWindow` 保存：

```text
cycle_id
instance_id
starts_at
ends_at
settlement_available_at
```

窗口采用左闭右开 `[starts_at, ends_at)`。实例 ID 由周期稳定 ID 和调度窗口键组成，例如：

```text
cycle.daily_reset@20260713
```

`settlement_delay` 可以让窗口结束后延迟一段时间再进入结算队列。它适合等待跨服数据归集，不应用来
模拟业务执行耗时。

## 漏执行补偿

每个周期显式选择 `CatchUpPolicy`：

- `ALL`：按时间顺序补齐全部遗漏窗口，并按 `maximum_backfill_per_scan` 分批推进游标。
- `LATEST`：只生成截至扫描时间最新的一个到期窗口，游标直接推进到本次扫描时间。
- `DISCARD`：不生成历史工作项，只推进游标。

游标记录的是“已经扫描到哪里”，不是“机器上次什么时候启动”。发现工作项和 CAS 推进游标位于
同一个 SQLite 写事务，不能出现游标已前进但工作项未写入的状态。

## 持久化执行队列

`PersistentCycleService` 提供：

```text
initialize_cursor
discover
claim
heartbeat
complete
retry
fail
requeue_failed
require_work
```

工作项状态为：

```text
pending -> running -> completed
                   -> pending  延迟重试
                   -> failed   永久失败
failed  -> pending             显式重新排队
```

`claim()` 可以领取到期的 pending 工作，也可以接管租约已经过期的 running 工作。每次领取都会增加
`attempt_count`。租约有效期间其他 worker 不能接管；心跳、完成、重试和失败只能由当前租约持有者
执行，租约过期后原 worker 也不能继续提交状态。

所有失败重排都要求显式 `retry_at`，系统不会在底座中暗藏指数退避或最大次数。具体退避策略属于
worker 组合层，永久失败后必须由管理流程显式调用 `requeue_failed()`。

## 业务防重

每个工作项拥有确定性事务 ID：

```text
SHA-256(scope_id | cycle_id | instance_id)
```

业务结算必须直接使用 `work.transaction_id` 作为奖励或经济事务 ID：

```python
work = cycles.claim(
    "worker.settlement",
    logical_time=logical_time,
    lease_duration=timedelta(minutes=5),
)

if work is not None:
    rewards.settle(
        RewardSettlement(id=work.transaction_id, ...),
        ...,
    )
    cycles.complete(
        work.transaction_id,
        "worker.settlement",
        logical_time=logical_time,
    )
```

奖励事务与工作项完成标记不强行放进同一个数据库事务。若进程在奖励提交后、标记完成前崩溃，租约
到期后工作会被重领；相同事务 ID 让现有奖励结算防重安全重放，然后再完成工作项。这解决了最危险
的崩溃窗口，同时不让周期底座依赖某一种业务结算模型。

## 内容包接入

`ContentPackage.cycles` 声明周期，`ContentAssembler` 会：

1. 登记周期稳定 ID 所有权。
2. 校验调度类型是否已有处理器。
3. 审计自定义调度类型的内容包依赖。
4. 把定义和处理器身份纳入内容指纹。
5. 返回已经冻结的 `runtime.cycles` 和 `runtime.cycle_engine`。

不同世界包可以声明任意数量的周期。周期名称和展示仍由世界皮肤或业务消息决定，稳定 ID 不直接
展示给玩家。

## 不属于本底座

- 每日任务、周榜、签到、赛季、活动和城主结算的具体规则。
- 某类工作应该发放什么奖励或修改哪些聚合。
- worker 的启动方式、扫描频率、退避算法和告警渠道。
- QQ 文案、按钮、后台管理页面和人工审批流程。
- 旧项目定时任务迁移与历史周期补算。
