# 权益凭证与兑付底座说明

## 封板版本

```text
grant.foundation.v1
```

奖励底座负责“怎么发”，权益底座负责“凭什么领”。兑换码、小游戏回执、活动发放、系统补偿和旧资产等价迁移最终都转换为账号权益，再调用统一奖励结算。

## 统一流程

```text
活动定义 + 奖励方案版本
          ↓
代码 / 签名回执 / 直接权益 / 迁移清单
          ↓
校验账号、时间、状态、账号额度、活动总额度和凭证次数
          ↓
构造稳定 RewardSettlement
          ↓
权益消费 + 奖励到账 + 防重事务 + Outbox 同一事务提交
```

`GrantCampaign` 只保存稳定的 `offer_id` 和 `offer_version`。具体游戏的可信内容层负责把奖励方案解析为 `GrantRewardBundle`；玩家输入不能携带任意奖励内容。

## 支持场景

- 单码单次：活动账号额度和总额度都为 1。
- 公共码：同一个代码可以按账号领取，受账号上限约束。
- 限额码：在账号上限之外再限制全服总领取量。
- 小游戏回执：绑定发行者、账号、活动、场次、nonce、结果摘要和有效期。
- 活动与补偿：预先向指定账号批量发放权益，领取时仍走统一兑付。
- 旧资产迁移：按批次、旧主体和旧资产唯一登记版本化等价映射。

## 凭证安全

兑换码经过 NFKC、大小写和分隔符归一化后，使用服务端密钥计算 HMAC-SHA256。数据库只保存摘要，不保存明文代码。代码密钥不能写入活动元数据、数据库或日志。

小游戏使用 `GrantProof`。签名覆盖：

```text
issuer_id
receipt_id
campaign_id
account_id
nonce
issued_at / expires_at
payload_digest
```

底座使用常量时间比较验证签名。小游戏服务端应签署经过服务器判定的结果摘要，不能让客户端直接决定奖励档位。

## 防重与并发

- 活动、代码、外部回执、批量权益和迁移清单支持同身份同内容幂等重放。
- 同身份不同内容抛出 `TransactionMismatch`。
- 兑付命令 ID、权益 ID、奖励结算 ID 和内容指纹全部稳定。
- SQLite 写事务使用 `BEGIN IMMEDIATE`，额度检查与计数写入串行完成。
- 活动总额度、账号额度和凭证次数都在奖励提交前重新检查。
- 已存在兑付记录时必须同时重放出相同奖励凭据，否则视为数据损坏。

## 原子边界

`PersistedRewardSettlementService.settle_in_uow()` 允许权益服务持有唯一外层工作单元。在同一个事务中完成：

```text
写入奖励领域新快照
写入奖励领取记录
写入 committed_transaction
写入 Outbox
标记权益已兑付
增加凭证使用次数
写入权益兑付凭据
```

奖励规划、余额、库存或并发校验任一失败时，新增的代码权益、凭证次数和所有奖励变化一起回滚。

## 迁移清单

`MigrationManifestEntry` 使用 `(batch_id, legacy_subject_id, legacy_asset_id)` 唯一定位旧资产，同时保存：

- 等价映射版本。
- 目标内部账号。
- 对应权益 ID。
- 原始数据摘要。
- 结构化来源数据和导入时间。

迁移清单不读取旧数据库结构，也不猜测旧昵称与新账号关系。业务必须先完成账号归属确认，再提交不可变清单。

## 撤销

- 活动可以暂停或撤销，状态更新使用版本 CAS。
- 代码和签名凭证可以撤销。
- 未领取权益可以撤销。
- 已兑付权益不能通过本底座反向删除玩家资产；需要回收时必须建立独立、可审计的资产事务。

## 数据表

```text
grant_campaign
grant_credential
grant_entitlement
grant_redemption
migration_manifest
```

## 不属于本底座

- 活动参与条件和玩家分群查询。
- 具体奖励方案内容与数值。
- 小游戏本身的分数判定。
- 旧账号如何绑定新账号。
- 玩家展示文案、按钮和管理后台页面。

这些上层能力只能发行或兑付权益，不能绕过统一奖励结算直接写资产。
