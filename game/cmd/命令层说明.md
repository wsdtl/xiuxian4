# 正式游戏命令层

`game/cmd/` 只负责把消息命令连接到正式游戏业务，不承载规则、数据库实现或平台私有协议。

## 注册命令

```python
from launch.adapter import Depends

from ..command import GameCommand
from ..dependencies import current_character
from . import service


@GameCommand.handler(cmd=("背包", "行囊"))
async def show_inventory(character=Depends(current_character)) -> None:
    await service.show_inventory(character)
```

- `priority=100` 和 `block=True` 已由 `GameCommand` 提供默认值。
- 命令默认使用 `access="player"`，要求当前身份已经创建角色。
- 创建角色、查询建档状态等公开入口必须显式写 `access="public"`。
- 多个别名写在同一个 `cmd` 元组中，不能复制多个处理函数。
- 需要从全局通栏进入的命令通过 `intent_ids=(...)` 与稳定回复意图绑定；没有正式命令就不能注册通栏链接。

## 公共依赖

- `current_identity_evidence`：只转换当前消息身份。
- `current_character`：只解析当前账号和角色，不读取其他领域。
- `current_character_overview`：按需读取角色、背包、装配、账本、世界和行动快照。

依赖结果在单条消息的依赖链中自动缓存。普通命令应优先使用最轻量的依赖，不能为了方便统一读取
`current_character_overview`。

## 访问守卫

`access_guard.py` 在业务回调执行前检查 `GameCommand` 的访问元数据。未建档用户访问默认玩家命令时，
只返回一次创建角色入口；公开命令不受影响。后续战斗、休息或活动状态守卫由对应系统独立注册，不能
全部堆进角色守卫。

## 发送回复

业务只构造 `message.Message`，再调用：

```python
await send_game_reply(reply)
```

`send_game_reply()` 是正式游戏的统一回复流水线。对于已建档玩家的 `DocumentMessage`，它会只读加载
最新人物设置、全服活动热点、有效未读通知数量和待领取行动数量，再由 `GameReplyComposer` 统一生成：

```text
人物头
活动通栏（没有热点时省略）
提醒通栏（没有待办时省略）
组件业务正文
附加区
按钮
```

组件不能调用 `inline_section()` 占用全局通栏，也不能手写彩色 Header。组件提供的普通 Header 只用于
未建档或身份异常等公共回复；已建档玩家的 Header 会统一替换为“境界短名 + 名称 + 等级”。活动通栏
最多显示两个可点击短名，其余热点显示为 `+N`；提醒通栏只显示个人数量。两者都是只读操作，不会自动
参与活动、已读通知、领取奖励或推进行动。

`world_events` 与 `world_events 活动实例ID` 是活动通栏使用的内部入口，不放入公开命令列表。具体活动
通过 `game.rules.activity.register_global_activity()` 和稳定 `entry_intent_id` 接入，不能修改回复装饰器。

图片消息不强行拼接文本外框。最终发送仍由公共驱动器管理器完成，游戏回复层不处理 QQ 私有 payload。

## 边界

- `__init__.py` 只放装饰器、命令参数和一次 service 调用。
- 文本解析使用小型纯函数，解析结果再交给 service。
- 危险操作由所属业务实现一次性确认凭证，命令层不保存临时业务状态。
- service 通过应用服务操作数据库，组件不能直接修改快照。
