# 伙伴正式业务

## 唯一职责

`game.features.companion` 是伙伴领域的唯一写入口，协调万灵引消费、秘境状态、固定踪迹、捕获战斗、
名册容量、六套配装的独占引用和永久放生。命令组件不能直接修改伙伴、库存、角色或配装快照。

## 状态所有权

- `game.companion.roster`：伙伴实例、玩家引用 `C1/C2`、捕获图鉴和配装引用。
- `game.companion.sanctuary`：一次秘境的来源世界、有效期、三条固定踪迹、锁定目标和尝试次数。
- 角色、库存、配装、行动与探险仍由各自领域拥有；伙伴业务只在联合事务中读取或协调更新。

开启秘境时，扣除一枚万灵引、创建或更新秘境、初始化空名册和保存幂等回执必须在同一工作单元提交。
捕获时，角色战后资源、名册新增实例、秘境终态、公开战报和幂等回执必须共同提交。任一步失败都回滚，
不能出现“物品已扣但秘境未开”或“战斗胜利但伙伴未入册”。

## 战斗边界

伙伴使用公共 Ability、Effect、Trigger、AI、目标系统与战报，不维护第二套战斗公式。组合根提供
`PlayerBattleLineupProjector`，把“角色当前配装 + 该配装绑定伙伴”投影为统一玩家阵容。探险、切磋、
多次元灾厄和伙伴秘境都使用这一入口。

伙伴是独立友方实体，拥有自己的速度、行动、资源、状态和事件。普通战斗结束后，伙伴的临时血气、
灵力与正负面状态不写回名册；角色资源是否持久化继续由具体战斗模式决定。切磋保持双方完全无损。

## 稳定约束

- 名册上限 30，开启秘境时预留一个容量位置；满员时不扣万灵引。
- 一套配装最多绑定一只伙伴，同一伙伴不能同时绑定多套配装。
- 放生永久删除实例并清理全部配装引用，不发放收益，但保留物种捕获记录。
- 开启秘境后跃迁不会刷新踪迹；玩家必须回到来源世界才能继续追踪。
- 踪迹首次选择后永久锁定，战败只增加尝试次数，不改变品质、资质、特性或战斗种子。
- 所有写操作使用消息证据组成幂等 ID；放生、放弃和跨配装转移还必须校验确认时 revision。
