# 统一战报服务

## 边界

- 战斗核心只产生 `RuleEvent`，不导入网页、QQ 或战报存储。
- 探险、灾厄、切磋、决斗和以后新增的模式只提交统一 `BattleReportDraft`。
- 战报服务保存摘要、兼容摘要状态和与 `BattleTrace` 一一对应的转场序列，不反向参与战斗结算。
- HTTP 组件只把已保存事实翻译成人类可读文字。

## 存储

- `battle_report` 保存模式、世界皮肤版本、内容指纹、公开摘要和保留期。
- `battle_report_segment` 保存一份报告中的连续战斗片段。
- 片段使用紧凑 JSON、`zlib level 6` 和 SQLite `BLOB`。
- 编码版本为 3；每个转场保存动作前后完整 BattleFrame、动作/AI 决策、请求目标、实际目标和该转场的全部规则事件。
- BattleFrame 保存回合、行动、状态、修订号、当前行动者、行动顺序、失活实体、胜方、行动进度及参与者快照。
- 参与者快照包含属性、资源、正负面效果、层数、剩余回合、冷却、招式、触发器、拦截器和目标限制。
- 新格式不重复保存平铺事件；读取接口提供由转场派生的 `events`，兼容旧的网页和检查代码。
- 当前全部战斗事件必须登记可读翻译；新增事件没有翻译时，覆盖测试直接失败。
- 实体 ID 在编码时改写为 `p0`、`p1` 等报告内局部代号，公开页不会泄漏角色、账号、QQ 或资产身份。
- `report_id + segment_id` 负责幂等；公开地址只使用随机不可枚举的 `share_id`。

## 保留期

- 完整片段保留 7 天。
- 第 8 至 30 天只保留结算摘要。
- 30 天后整份删除，公开地址返回 404。
- 清理任务每天运行一次，重复执行结果一致。

## 正式内容覆盖

- 七十二把正式武器逐把执行并检查战报事件。
- 四十八类装备词条参与生成巡检，七十五个装备触发器逐个实际激活。
- 十二套套装逐套装配并校验全部贡献引用。
- 三十二种敌人行为能力逐个执行，首领阶段引用必须落在已覆盖行为集合。
