# 统一战报服务

## 边界

- 战斗核心只产生 `BattleTrace`、状态帧与 `RuleEvent`，不导入网页、QQ 或战报存储。
- 七种战斗模式只提交核心轨迹和参战者身份；唯一 `BattleReportBuilder` 负责生成 `BattleReportDraft`。
- 战报服务只保存、读取和清理已发生事实，不反向参与战斗、奖励或资源结算。
- `BattleReportPresenter` 在后端把冻结事实完整解释为公共展示节点；HTTP 只返回文档，Web 只渲染节点。
- Web 不识别事件类型、内容 ID、武器、装备、状态或资源词，也不自行拼正文、推断状态差异或猜测结局。

## 参战者投影

- 不存在报告级全局皮肤。每名参战者独立冻结 `projection_kind/id/version` 和本场实际用到的词表。
- 角色按开战时自己的 `world_id` 解释武器、装备、能力、资源与状态；铭刻最终名称在开战时覆盖并冻结。
- 敌人、组队首领和灾厄按自身来源世界解释，不读取玩家、队长或战场的世界。
- 伙伴不属于归航的跨界同化对象，永久按实例 `origin_world_id` 保留来源世界的名称、品阶、资源与能力语义。
- 同一事件涉及不同来源时分别读取双方词表，例如资源转移可以同时出现“灵力”和“同步”。
- 战报保存完成后不再读取实时世界皮肤；角色跃迁、内容更新或铭刻变化都不能改写历史战报。

## 存储

- `battle_report` 保存模式、内容指纹、公开摘要、总压缩体积和保留期。
- `battle_report_segment` 保存一份报告中的连续战斗片段。
- Codec v4 使用紧凑 JSON、`zlib level 6` 和 SQLite `BLOB`；不读取或兼容旧战报结构。
- 每个片段只保存一次参战者清单和冻结词表；每个转场保存动作前后完整 `BattleFrame`、动作与 AI 决策、请求目标、实际目标和全部规则事件。
- 状态帧保存回合、行动、状态、修订号、行动顺序、失活实体、胜方、行动进度，以及属性、资源、效果实例、层数、剩余回合、冷却、招式、触发器、拦截器和目标限制。
- 当前全部战斗事件必须在后端登记玩家可读语义；新增事件未登记时测试和公开展示直接失败，不能交给 Web 猜测。
- 实体、资产和效果实例 ID 在编码时改写为报告内局部代号，公开页不会泄漏角色、账号、QQ 或资产身份。
- `report_id + segment_id` 负责幂等；公开地址只使用随机不可枚举的 `share_id`。

## 保留期

- 完整片段保留 7 天。
- 第 8 至 30 天只保留结算摘要。
- 30 天后整份删除，公开地址返回 404。
- 清理任务每天运行一次，重复执行结果一致。

## 正式内容覆盖

- 正式武器逐把执行并检查战报事件；新增武器只注册核心事实和后端语义，不得修改网页识别具体武器。
- 装备词条、装备触发器和套装贡献必须经过引用与实战覆盖。
- 敌人行为与首领阶段逐项执行，来源世界投影必须完整覆盖实际用到的内容 ID。
- 主测试同时验证太玄、魔法、星环三套资源语义，以及伙伴来源世界不会被主人或战场覆盖。
