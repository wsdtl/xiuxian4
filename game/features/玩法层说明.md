# 正式玩法层

`features` 保存建立在公共核心和具体规则之上的正式玩法。每个玩法按领域建立独立子目录，负责组织
跨角色、库存、世界、掉落和奖励等用例，但不拥有这些领域的底层状态，也不导入 QQ、消息协议或命令
注册。

目录边界固定如下：

- `game/content/catalog/<domain>`：稳定名录、概率和内容定义。
- `game/rules/<domain>`：数据库、机器时间和消息协议无关的纯计算规则。
- `game/features/<domain>`：由组合根注入基础设施的跨领域玩法协调。
- `game/cmd/<中文组件>`：命令参数、按钮和玩家展示。

玩法层可以保存运行保护，例如单次追赶批次数和调度扫描上限；这类值只控制一次任务最多处理多少工作，
不能改变玩家收益、掉率、成长、恢复或活动展示结果。凡是会改变游戏平衡或正式内容表现的数字，仍必须
归入 `game/content/catalog/<domain>`。

玩法层不能直接导入 `game.core.persistence`。SQLite 数据库、快照仓储、聚合类型和持久化服务必须由
`game/app.py` 创建后注入；玩法只接收所需能力和存储键，不自行实例化基础设施。新增玩法不得在
`game/` 根目录创建同名业务文件，也不能把业务实现塞进 `game/app.py` 或命令组件。

完整业务能力台账、新业务准入问题和目录模板见《业务框架与功能规划》。`catalog.py` 只登记已经落地的
正式业务，不登记设想或空占位。

当前 `exploration` 负责持续探险的跨域原子结算，`economy` 负责统一估价后的归航回收、归航市场、
跨玩家库存转移和归航库拆账。`sparring` 负责按接收角色隔离十分钟邀请、读取双方真实快照、执行
无损自动战斗并写入公开战报。`lottery` 负责单人单期唯一号码、环形号码开奖、奖池补贴、人数不足退票、归航库支出和个人中奖记录，
购票与开奖分别在独立的联合事务中提交。`draw` 负责在同一事务中扣除抽奖签、推进保底、结算物品或
货币奖励并保存幂等历史；奖项和概率仍归 `game/content/catalog/draw`。
`companion` 负责宠物秘境与捕获、地图人物赠礼与结交、通用名册、配装独占引用和告别事务；
伙伴进入其他战斗模式时统一通过玩家阵容投影组合，不复制探险、切磋或灾厄的战斗流水线。
`party` 负责三人队伍、邀请接力、成员关系、队长、站位和准备状态；当前不拥有多人战斗、奖励或掉落，
`party_battle` 在自己的事务边界内读取队伍快照，负责跨界组队首领、临时战斗投影、独立奖励和公开战报。
`breakthrough` 负责等级关隘、破境凭证、经验结算和资源恢复；`battle_report` 负责所有模式共用的公开战报。
`player` 负责身份到角色、跨域角色总览、个人设置和提醒读模型；`dimension_shift` 负责跃迁条件、凭证
扣除、真实世界更新和空间迁移。`game/app.py` 只保留这些业务的装配、异常映射和稳定转发入口。
`world_travel` 负责所有世界内地点移动和旧按钮上下文校验；探险与人物业务不得自行写世界位置。
`world_progress` 通过探险结算观察端口消费最小胜利事实，按角色、真实世界和区域独立累计行纪，沿用
通用奖励底座发放一次性阶段奖励，并维护可从权威快照重建的永久排名投影。探险不导入行纪业务，也不
保存行纪字段。
