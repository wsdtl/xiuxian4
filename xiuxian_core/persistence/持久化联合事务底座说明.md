# 持久化联合事务底座说明

## 封板版本

```text
persistence.foundation.v1
```

数据库结构版本为 `1`。代码分别通过以下常量读取：

```python
PERSISTENCE_FOUNDATION_VERSION
PERSISTENCE_SCHEMA_VERSION
```

## 一句话理解

规则层已经能算出一组全有或全无的新快照，持久化层负责保证这些结果在数据库里也只会出现两种
情况：

```text
全部提交
全部没有发生
```

库存写入成功但货币、角色、武器或领取记录失败的中间状态不能被其他请求看到。

## 分层边界

持久化分为两层：

```text
SQLite 通用基础设施
  结构版本
  工作单元
  CAS 聚合快照
  提交事务防重
  Outbox
  内容包激活指纹
  周期游标与租约工作队列

奖励结算持久化适配器
  读取各领域快照
  调用 RewardSettlementEngine
  条件写回变化聚合
  登记领取事务
  写入结构化事实

铭刻持久化适配器
  联合提交铭刻之羽消费与实例改名
  保存个人原名展示偏好
  使用提交事务防止重复消费
```

`xiuxian_core/gameplay/` 和 `xiuxian_core/account/` 不导入 `xiuxian_core/persistence/`。持久化适配器可以认识领域模型，领域模型不能
认识 SQLite、表名或 JSON 编码。

## 数据库结构版本

`SqliteDatabase.initialize()` 必须在业务服务开始接收请求之前执行。

- 空数据库会创建当前结构并写入唯一版本号。
- 重复初始化当前版本是安全的。
- 已经存在未知业务表但没有版本标记时拒绝启动。
- 版本号不同或当前结构缺表时拒绝启动。
- 不猜测旧表含义，不静默升级，不写兼容分支。

当前新项目尚未发布历史版本，因此不存在迁移器。未来真的发布结构版本后，迁移必须是独立、显式、
可审计的发布步骤，不能藏在正常启动中。

## 工作单元

`SqliteUnitOfWork` 使用独立连接。写工作单元执行 `BEGIN IMMEDIATE`，读工作单元执行
`BEGIN DEFERRED`。

离开上下文不会自动提交。只有显式调用 `commit()` 才会落盘，因此异常、提前返回和规则失败都会
回滚：

```python
with database.unit_of_work() as uow:
    # 多个仓储写入
    uow.commit()
```

SQLite 启用：

```text
foreign_keys = ON
journal_mode = WAL
synchronous = FULL
busy_timeout = 配置值
```

## 聚合快照与 CAS

`aggregate_snapshot` 用 `(aggregate_kind, aggregate_id)` 唯一定位一个聚合，并分别保存
`revision`、编码版本和结构化负载。

更新语句必须满足：

```sql
WHERE aggregate_kind = ?
  AND aggregate_id = ?
  AND revision = expected_revision
```

并且新 revision 必须恰好增加 `1`。没有命中一行就抛出 `ConcurrencyConflict`，整笔工作单元
回滚。聚合 ID 由组合根选择，可以按账号、世界、币种或其他明确边界分片，不能在仓储中根据昵称
猜测。

当前已登记聚合类型：

```text
snapshot.inventory
snapshot.ledger
snapshot.character
snapshot.weapon
snapshot.reward_claim
```

通用仓储不限制以后增加新聚合，但新类型必须拥有明确 revision 和白名单编码。

## 结构化 JSON

快照不使用 `pickle`，也不根据数据库字符串动态导入 Python 类。`StructuredJsonCodec` 只还原启动
时明确登记的 dataclass 与 Enum，并原生支持：

```text
datetime 时区
tuple
frozenset
mapping
有限浮点数
```

未知类型、未知编码版本、负载根类型错误、负载 revision 与行 revision 不一致都会被视为数据损坏。
名称和世界皮肤仍然不能成为业务键。

## 数据库级防重

`committed_transaction.transaction_id` 是数据库唯一键。正式奖励提交同时保存：

```text
奖励内容指纹
领取作用域
结构化结算凭据
逻辑提交时间
```

重复请求必须同时满足：

- 数据库事务表存在相同指纹。
- 对应领取快照可以重放相同奖励。
- 两处保存的结算凭据完全一致。

只存在其中一处会被视为 `CorruptPersistenceData`，不会偷偷补写另一半。

## 内容包激活

`content_activation` 保存每个激活槽当前使用的内容指纹、包版本集合、战斗配置和 revision。普通启动
只能验证完全一致的 `ContentAssemblyReport`；内容变化必须使用旧 revision 与旧指纹执行显式 CAS
切换。数据库不会在启动时自动接受一套新规则。

## Outbox

所有 `RuleEvent` 与领域快照、领取记录和提交事务在同一个数据库事务中写入 `outbox_event`。
消息发送、统计、审计和网页更新只能在提交完成后读取 Outbox，不能在数据库事务中直接访问 QQ 或
外部网络。

Outbox 使用 `(transaction_id, sequence)` 唯一定位事件。发布成功后显式写入 `published_at`。
投递语义是至少一次：外部消费者必须使用这个稳定事件身份防重，不能假设网络发送与数据库标记
之间绝不会中断。

## 周期工作队列

`cycle_cursor` 保存每个作用域、每个周期已经扫描到的逻辑时间；`cycle_work_item` 保存到期窗口、
确定性事务 ID、执行状态、租约、重试时间和错误。窗口入队与游标 CAS 推进处于同一个写工作单元，
服务重启不会漏掉已经推进的窗口。

工作项支持到期领取、租约过期抢占、心跳续租、延迟重试、永久失败和显式重新排队。所有时间由调用
方传入并统一按 UTC 保存，持久化层不读取机器当前时间。完整规则和业务防重约束见
`xiuxian_core/gameplay/cycles/时间与周期底座说明.md`。

## 奖励持久化流程

`PersistedRewardSettlementService.settle()` 的固定步骤：

1. 开始写工作单元。
2. 读取库存、账本、角色、武器和领取作用域快照。
3. 检查数据库事务防重记录。
4. 调用纯规则 `RewardSettlementEngine`。
5. 使用原 revision 条件写回所有变化聚合。
6. 插入提交事务与结构化凭据。
7. 顺序写入全部 Outbox 事实。
8. 显式提交。

任意步骤失败都会恢复规则随机游标并回滚数据库。规则失败以 `RuleOutcome.failure` 返回；数据库损坏、
结构版本错误和并发写冲突使用明确持久化异常，不伪装成游戏规则失败。

## 新聚合初始化

`initialize_snapshot()` 只用于新账号或新世界聚合创建。库存、账本、角色、武器和领取作用域会在
同一个事务中插入，其中任何一行已经存在都会让整批初始化回滚。旧数据导入不属于这个入口。

铭刻名称直接位于库存中的独立物品实例，不增加第二份资产聚合。个人原名展示偏好使用
`snapshot.inscription_preference`，由 `PersistedInscriptionService` 显式初始化和条件更新。

## 应用配置

`.env`：

```dotenv
DATABASE_PATH=data/xiuxian4.db
DATABASE_BUSY_TIMEOUT_MS=5000
```

组合根显式初始化：

```python
from launch import config
from xiuxian_core.persistence import SqliteDatabase

database = SqliteDatabase(
    config.database.path,
    busy_timeout_ms=config.database.busy_timeout_ms,
)
database.initialize()
```

`launch` 生命周期不自行组装 Gameplay 目录或奖励引擎。正式游戏组件完成组合后，应通过启动回调
调用数据库初始化，并把同一个数据库对象注入持久化服务。

## 当前容量边界

SQLite 适合单机器人、单服务进程。项目已有单实例锁，SQLite 写事务也会串行化磁盘提交。聚合 ID
仍然必须按真实所有权合理分片，避免把全服库存或无关账号领取记录塞进一个快照。

如果未来需要多进程或多节点，保留 `SnapshotRepository`、CAS、事务防重和 Outbox 语义，新增
PostgreSQL 适配器；不能让上层业务改成直接执行数据库方言。

## 不属于本底座

- 物品、角色、经济或奖励业务规则。
- 内容包加载和世界皮肤注册。
- 日、周、赛季和活动的具体业务结算。
- QQ 消息渲染与发送。
- 历史版本迁移和 xiuxian3 数据导入。
- Outbox 具体消费者和外部消息代理。
