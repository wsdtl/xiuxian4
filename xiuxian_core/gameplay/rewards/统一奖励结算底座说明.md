# 统一奖励结算底座说明

## 封板版本

```text
reward.foundation.v1
```

代码可通过 `gameplay.rewards.REWARD_FOUNDATION_VERSION` 读取版本。

## 一句话理解

业务系统只负责提交一份已经算好的奖励清单：

```text
货币 250
物资 5
独立物品 1
角色经验 120
武器经验 150
解锁特征 1
```

统一结算层负责让这些奖励一起预检、一起成功或一起失败，并生成一张稳定凭据。它不决定奖励
数量、掉落概率、保底、税率或活动规则。

## 为什么需要这一层

没有统一结算时，一个任务可能依次执行：

```text
先发钱成功
再发物品成功
最后增加经验失败
```

玩家会拿到半份奖励，重试又可能重复拿到前半份。`RewardSettlementEngine` 在现有库存、账本、
角色和武器引擎之上建立一个规则工作单元：所有领域只产生候选快照，最后一步成功后才把完整结果
交给持久化层。

## 支持的标准奖励

- `CurrencyReward`：通过唯一发行账户发放货币。
- `StackItemReward`：发放可堆叠物资并记录来源批次。
- `InstanceItemReward`：发放拥有独立资产 ID 的物品实例。
- `CharacterExperienceReward`：增加指定角色成长轨道经验。
- `CharacterFeatureReward`：解锁永久特征。
- `CharacterProgressionReward`：解锁成长轨道。
- `WeaponExperienceReward`：增加指定武器实例经验。

同一结算中指向同一角色的操作会合并成一个角色事务；同一武器的多条经验会先求和，再执行一个
武器事务。库存和账本也分别只执行一个领域事务，避免同一奖励无意义地增加多次 revision。

## 快照不是新的数据所有者

`RewardSettlementSnapshot` 只是一次结算借用的视图：

```text
InventoryState 仍属于库存域
LedgerState 仍属于经济账本
CharacterState 仍属于角色域
WeaponState 仍属于武器域
RewardClaimState 只属于奖励领取记录
```

结算层不会重新定义余额、背包、角色或武器表，也不会允许这些领域反向依赖奖励模块。

领取记录通过 `claim_scope_id` 明确分片，通常使用内部账号 ID，也可以使用组织或系统业务主体 ID。
`actor_id` 表示谁发起或授权结算，两者不能混用。这样系统代发奖励时仍归入玩家领取记录，不会让
全服无关玩家因为共享一个领取 revision 而互相阻塞。

## 预检与正式结算

`preflight()` 和 `settle()` 使用同一条规划、校验和领域执行路径。

- `preflight()` 返回 `RewardSettlementPreview`，只包含结算凭据和结构化事实。
- 预检不登记领取、不暴露候选领域快照，并恢复规则随机游标。
- `settle()` 返回新的完整快照和领取记录，原快照始终不变。
- 正式提交前仍会重新检查所有 revision，预检成功不代表可以绕过并发校验。

这样 QQ 可以先展示奖励确认或选择结果，但不能误把预检结果当成正式资产。

## 并发预期

`RewardExpectations` 明确声明本次结算读取的版本：

```text
当前 claim_scope 的领取记录 revision
库存 revision
参与发币的账本账户 revision
参与成长的角色 revision
参与成长的武器 revision
```

只要求本次奖励真正接触的领域和对象。缺少 revision、提供无关 revision 或实际版本已经变化都会
拒绝整笔结算，不会覆盖其他业务刚提交的结果。

## 防重复领取

`settlement_id` 必须是全局稳定的业务领取 ID，例如：

```text
quest:daily-2026-07-13:account-a
boss:boss-run-883:character-a
mail:mail-912:attachment
```

奖励指纹包含领取者、来源、完整奖励清单和业务元数据，但不包含 revision：

- 相同 ID、相同奖励重试，直接返回原凭据，标记 `replayed=True`。
- 相同 ID、不同奖励重试，返回 `reward.settlement_mismatch`。
- 首次失败不会写入领取记录，修正并发版本后仍可使用原 ID 重试。

数据库实现必须对 `settlement_id` 建立唯一约束，内存中的指纹记录不能替代数据库约束。

## 解锁重复策略

永久特征和成长轨道默认使用 `DuplicateUnlockPolicy.IGNORE`。玩家已经拥有时，该行凭据标记为
`SKIPPED`，不会让其他奖励失败，也不会重复增加角色 revision。

必须严格校验的特殊奖励可以使用 `REJECT`。同一奖励清单里重复出现的相同解锁也遵守该策略。

## 来源与凭据

物品奖励自动生成确定性的 `SourceReceipt`：

```text
receipt_id = settlement_id + reward_index
source_kind = 奖励业务类型
source_id = 奖励业务实例
```

`RewardReceipt` 按原奖励顺序保存每一行的类型、目标、内容、数量和发放状态，同时记录实际调用的
领域事务 ID。QQ、网页、日志和统计可以翻译凭据与 `RuleEvent`，不能从展示文本反推资产变化。

## 扩展奖励类型

`RewardPlannerRegistry` 在启动期登记“奖励声明如何转换为现有领域操作”。扩展处理器只能通过
`RewardPlanBuilder` 提交库存、账本、角色或武器操作，不能直接修改快照。引擎组装后注册表冻结，
避免运行时热改奖励语义。

新增一种掉落包装、活动奖励或世界包内容通常只需要新增奖励声明和规划器。只有引入全新的资产域
时，才需要扩展外层结算工作单元。

## 失败边界

任意规划器或领域失败时：

```text
货币不发行
物品不进入背包
角色和武器经验不增加
解锁不生效
领取记录不写入
规则随机游标回滚
```

当前底座保证规则层只返回“全部旧快照”或“全部新快照”。下一层持久化联合事务必须把这些新快照
和领取记录放在同一个数据库事务中提交，才能把规则原子性完整延伸到数据库。

## 不属于本底座

- 掉落表、随机权重、保底和奖励倍率。
- 任务、首领、活动、邮件和成就的完成条件。
- 物品价格、税率、交易撮合和货币兑换。
- QQ 文案、按钮、图片和网页展示。
- 数据库表、仓储、事务隔离和失败重试队列。

这些业务可以生成 `RewardSettlement`，不能直接修改各领域快照。

## 重新审核条件

普通新增奖励内容或规划器不需要重开底座。只有以下情况需要更新封板版本：

- 新资产域无法接入现有规划和候选快照模型。
- 业务要求成功提交部分奖励而不再全有或全无。
- 防重身份不再能由稳定结算 ID 和内容指纹表达。
- 预检与正式结算必须使用不同规则语义。
- 需要改变已发布凭据或结构化事实的实际语义。

## 验证

```powershell
.venv\Scripts\python.exe -X utf8 -B test\reward_settlement_test.py
```

测试覆盖货币、堆叠物、独立物品、角色经验、武器经验、永久特征、成长轨道、重复解锁跳过、预检、
领取防重、同 ID 内容冲突、revision 冲突，以及最后一个领域失败时前置候选结果全部回滚。
