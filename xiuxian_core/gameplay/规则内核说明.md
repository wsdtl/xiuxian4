# Gameplay 规则内核

`xiuxian_core/gameplay/` 是游戏业务最底层的共同语言。它不知道 QQ、数据库、角色表、背包表和具体世界名称。

## 一句话理解

```text
Ability 表示“要做什么”
Effect 表示“产生什么变化”
Modifier 表示“怎样修改属性”
Tag 表示“它具有什么性质”
SkinPack 表示“某个皮肤版本里玩家看到它叫什么”
SkinCatalog 表示“项目拥有的任意数量皮肤和历史版本”
```

## 为什么先建设这一层

武器、技能书、药物、体质、敌人和宗门增益都可能改变属性或授予能力。如果每个组件自己实现效果，最后一定会出现多套伤害、恢复、上限和叠层规则。

规则内核只提供一次统一计算。以后物品系统负责“玩家拥有什么”，战斗系统负责“在什么战斗规则中执行”，经济系统负责“如何定价和交换”，它们都不重新解释效果。

## 稳定 ID 与世界皮肤

业务保存稳定 ID：

```text
item.main_hand.fast_001
```

皮肤决定玩家看到的名称：

```text
修仙皮肤：青锋剑
魔法皮肤：秘银短杖
```

业务判断永远不能搜索“青锋剑”。世界皮肤切换后，角色资产和规则不需要迁移。

`SkinCatalog` 不设置世界数量上限，也没有主皮肤、备用皮肤等固定槽位。每套皮肤使用稳定 `skin_id`，每次更新形成新的 `version`。运行时可以读取最新版本，历史消息也可以按原版本还原。

## 标签

标签使用父子层级：

```text
item.equipment.main_hand
weapon.tempo.fast
effect.damage.physical
state.control.stun
```

拥有 `effect.damage.physical.bleed` 的内容，也会满足 `effect.damage.physical` 条件。

## 属性计算

所有属性使用固定顺序：

```text
基础值
→ 本地固定增量
→ 本地比例
→ 全局固定增量
→ 全局比例
→ 最终倍率
→ 上下限
```

`AttributeSnapshot.breakdowns` 保存每层明细，数值异常时可以直接查出来源。

## Effect

一个 Effect 可以组合多个原子操作：

- 修改属性。
- 改变当前资源。
- 临时授予标签。
- 临时授予 Ability。
- 临时授予 Trigger。
- 临时授予伤害干预器。
- 转移双方资源。
- 净化、消耗层数和改变持续时间。
- 修改 Ability 冷却。

Effect 明确定义持续时间和叠层方式。瞬时 Effect 不能偷偷携带永久属性修改。

新增普通 Effect 只需要增加定义。真正出现全新机制时，新增一个有类型的数据类和对应处理器，通过 `EffectOperationHandlers.register(...)` 注册，不需要修改 EffectEngine。

## Ability

Ability 统一负责：

- 检查使用者和目标标签。
- 检查并扣除资源。
- 按顺序施加 Effect。
- 设置冷却。
- 产生结构化 RuleEvent。

战斗攻击、药物使用和装备主动能力可以使用同一执行链，但资产扣除、货币转账和掉落判定不属于 Ability。

## 固定执行阶段

规则按照固定阶段执行：

```text
PREPARE
→ PAY_COST
→ SELECT_TARGET
→ BEFORE_APPLY
→ RESOLVE
→ AFTER_APPLY
→ TURN_END
```

Ability 引用 Effect 时明确指定阶段。一个 Effect 同时提高血气上限并恢复血气时，会先安装持续属性修改，再按新的血气上限结算恢复，不依赖代码偶然顺序。

## RuleContext

每次规则执行都必须传入 `RuleContext`：

- `trace_id`：串起同一次执行产生的所有事实。
- `rule_version`：说明使用哪版规则。
- `Ruleset`：说明当前是普通战斗、首领或其他场景。
- `logical_time`：带时区的逻辑时间，不在内核里读取系统时间。
- `RandomSource`：显式随机源，相同种子和调用顺序可以重放。
- `phase`：当前执行阶段。
- `trigger_depth`：当前触发深度。

每条 `RuleEvent` 都固化 trace、规则版本、Ruleset、逻辑时间和阶段，日志不需要从外部猜测结算环境。

## 条件

条件是可注册的类型，不允许内容定义嵌入任意 Python 判断。当前内置：

- 标签条件。
- 属性阈值。
- 当前资源比例。
- Effect 层数。
- 事件字段比较。

条件引用的属性、资源和 Effect 会在规则注册表冻结前统一校验。

## Trigger

Trigger 监听 `RuleEvent`，条件成立后施加一个 Effect。Trigger 可以由持续 Effect 临时授予，因此装备、体质、状态和场景都能提供反震、吸血、击败触发等规则，而不需要修改战斗系统。攻击类触发必须监听 `combat.*` 专用事实，不能再用普通资源变化猜测是否受伤。

Trigger 明确规定：

- 监听什么事件。
- 拥有者必须是事件来源、事件目标或任意实体。
- 效果施加给拥有者、事件来源还是事件目标。
- 满足哪些条件。
- 独立触发概率；概率统一使用 `RuleContext.random`，可重放并在失败时回滚。
- 在哪个阶段执行。
- 单次执行最多触发多少次。

`Ruleset.max_trigger_depth` 限制连锁深度，单个 Trigger 还有独立次数上限。超过边界时整次纯规则执行失败，不返回半成品状态。

Ability 每施加一段 Effect，都会立即处理该段产生的 Trigger，再进入下一段。
因此第一段攻击触发的护盾、反击或状态能够影响第二段攻击。所有批次共享同一个
`TriggerSession`，触发次数上限和事件序号仍按整次 Ability 计算。

## 失败协议

玩家可以预期的失败使用稳定错误码，例如：

```text
ability.not_owned
ability.cooldown_active
ability.owner_condition_failed
ability.target_condition_failed
resource.insufficient
effect.target_blocked
rule.recursion_limit
```

上层玩法调用 `GameplayExecutor.execute_ability(...)` 获得 `RuleOutcome`。成功时包含新状态和事件，失败时包含 `RuleFailure`；QQ 和网页按错误码选择自己的文案，不解析异常字符串。

## 结构化事实

规则执行产生 `RuleEvent`，例如：

```text
ability.started
effect.applied
resource.changed
ability.completed
```

QQ、网页和日志以后只翻译这些事实。新增普通效果时，不需要再给三个展示出口分别补逻辑。

## 当前边界

下列具体玩法尚未建设：

- 战斗奖励、自动战斗 AI 和玩法入口；伤害与战斗时间线已经由
  `xiuxian_core/gameplay/combat/` 承接。
- 货币账本与经济。
- 探险、首领、宗门和仙城。
- 数据库存储。

角色成长和规则实体投影已经由 `xiuxian_core/gameplay/character/` 承接，详见
[角色与成长底座说明](character/角色与成长底座说明.md)。物品资产、容器和物资事务已经由
`xiuxian_core/gameplay/inventory/` 承接，详见
[物品与物资底座说明](inventory/物品与物资底座说明.md)。武器、六槽装备和原子装配分别由
`xiuxian_core/gameplay/weapon/`、`xiuxian_core/gameplay/equipment/` 和 `xiuxian_core/gameplay/loadout/` 承接。其余系统随后建立在规则
内核之上，不能反向让 `xiuxian_core/gameplay/` 导入它们。

## 验证

```powershell
.venv\Scripts\python.exe -X utf8 -B test\gameplay_kernel_test.py
```

测试覆盖稳定 ID、父子标签、注册表冻结、任意数量和多版本世界皮肤、固定种子重放、执行阶段、条件、标准失败码、药物恢复、临时增益、攻击消耗、效果到期、触发深度保护和自定义原子操作注册。伤害、护盾、吸血与反伤由独立战斗内核测试覆盖。
