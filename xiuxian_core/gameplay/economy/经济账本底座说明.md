# 经济账本底座说明

## 封板版本

```text
economy.foundation.v1
```

代码可通过 `gameplay.economy.ECONOMY_FOUNDATION_VERSION` 读取版本。

## 一句话理解

经济账本只回答这些问题：

```text
钱属于哪个账户
其中多少仍然可用
一次资金变化是否守恒
这笔变化是否重复执行
失败后是否完整回滚
```

它不决定物品价格、奖励数量、税率、交易规则或玩家看到的货币名称。

## 货币定义

`CurrencyDefinition` 使用稳定英文 ID。`decimal_places` 只描述展示精度，所有余额、冻结金额和
流水金额都必须使用整数最小单位：

```text
decimal_places = 0：1 表示 1 原石
decimal_places = 2：1 表示 0.01 个展示单位
```

账本不接受 `float`，也不在结算时做小数舍入。经济政策层必须先把最终整数金额算好。货币目录
在 `LedgerEngine` 组装时冻结，运行期不能临时增加货币。

## 账户模型

每个 `LedgerAccount` 只保存一种货币。玩家、宗门、仙城、市场和中央系统的差别由
`owner_kind + owner_id` 表达，不需要为每种玩法复制一套余额表。

账户职责只有三种：

- `STANDARD`：玩家钱包、组织资金、地方或中央资金池等普通账户。
- `ESCROW`：市场、交易或挑战流程已经接管所有权的托管账户。
- `ISSUER`：该货币唯一的发行清算账户。

同一种货币只能有一个发行账户。普通账户和托管账户余额不能为负；发行账户余额不能为正。
发行后，发行账户记负数、接收账户记正数，两边始终相加为零。回收是相反方向，因此可以直接从
发行账户余额得到货币的历史净发行量。

## 复式流水

每次真实资金移动都会生成一个 `JournalEntry`。其中至少包含一条负分录和一条正分录，同一币种
所有 `LedgerPosting.amount` 之和必须为零：

```text
玩家付款 -100
商家到账  +80
仙城税金  +15
中央税金   +5
合计        0
```

拆账是一条不可分割的流水，不是先扣款再依次给各方加钱。流水记录事务、操作者、逻辑时间、
业务原因和结构化元数据，QQ 文本和网页只能翻译事实，不能反向参与结算。

## 标准操作

- `OpenLedgerAccount`：以零余额开立账户。
- `TransferFunds`：普通账户之间一对一转账。
- `SplitFunds`：从一个账户原子拆分到多个账户。
- `IssueFunds`：从唯一发行账户向普通账户发行货币。
- `RetireFunds`：把普通账户资金送回发行账户并退出流通。
- `PlaceFundHold`：在原账户冻结资金，但不转移所有权。
- `ReleaseFundHold`：明确释放冻结资金。
- `CaptureFundHold`：从冻结金额中扣款并原子拆给一方或多方。

普通转账不能接触发行账户，防止业务代码把“发钱”伪装成转账。托管已经发生所有权转移时，使用
普通转账把钱转入 `ESCROW` 账户；只是防止玩家重复花费时，使用 `FundHold`。

## 冻结与捕获

`FundHold` 保存业务类型、业务实例、金额、创建时间和可选过期时间。冻结不会改变账户总余额，
只会降低可用余额。捕获可以只扣一部分，剩余部分继续冻结，也可以一次拆分为货款、税金和分成。

过期只根据 `RuleContext.logical_time` 判断，不读取机器时间。下一笔成功事务会清理已经过期的预约
并产生 `economy.funds.released` 事实；失败事务不会顺便提交清理结果。

## 并发与防重

已有账户参与变更时，调用方必须在 `expected_revisions` 中提供当前账户 revision。任意账户已经被
其他事务修改，整笔事务返回 `economy.revision_conflict`，不会覆盖新状态。

`transaction_id` 是稳定防重键：

- 相同 ID、相同内容再次提交，返回当前快照并标记 `replayed=True`，不重复入账。
- 相同 ID、不同内容再次提交，返回 `economy.transaction_mismatch`。

内存快照保存防重摘要；数据库实现还必须为 `transaction_id` 建立唯一约束，不能只依赖应用进程。

## 原子边界

`LedgerEngine` 在事务私有副本中执行全部操作，成功后一次生成新 `LedgerState`。任一步失败时：

```text
余额不变
冻结不变
流水不增加
revision 不增加
防重记录不增加
规则随机游标回滚
```

未来玩家交易需要更外层的持久化联合事务，把账本结果和库存结果一起提交。账本不能自行调用库存，
库存也不能自行修改余额。

## 结构化事实

```text
economy.account.opened
economy.funds.transferred
economy.funds.split
economy.funds.issued
economy.funds.retired
economy.funds.held
economy.funds.released
economy.funds.captured
```

## 不属于本底座

- 商品参考价、商店售价、市场撮合和竞价。
- 税率、累进规则、地方与中央分税比例。
- 掉落货币数量、任务奖励和运营补偿规则。
- 物品库存、武器、装备和角色经验。
- 数据库表、缓存、消息协议和玩家可见文案。
- 跨币种兑换价格；兑换服务只能提交两种币各自平衡的明确账本方案。

这些上层系统负责计算“应该怎样分”，账本只负责验证并完整执行。

## 重新审核条件

新增货币、账户所有者类型、税种或市场玩法不需要重开底座。只有以下情况需要更新封板版本：

- 业务要求允许不守恒的流水。
- 一笔分录必须同时混合多种货币且无法拆成各币种独立平衡。
- 现有冻结、部分捕获和托管模型无法表达新的资金占用。
- 账户 revision 与事务防重无法满足新的并发模型。
- 需要改变已发布结构化事实的实际语义。
